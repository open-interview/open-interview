> "The hardest thing about distributed systems isn't concurrency—it's uncertainty. When a payment succeeds but the inventory reservation fails, how does your system restore sanity? The answer lies in thoughtful saga design."

Building a retail order platform that handles thousands of concurrent orders across multiple tenants requires more than well-crafted APIs. It demands a battle-tested approach to distributed transactions, where failures are not anomalies but expected events. In this post, I'll walk through the architectural decisions and implementation details of a saga-based order system using CQRS and event sourcing.

## Orchestration vs. Choreography: The Case for Coordination

In distributed systems, you face a fundamental choice: let services communicate directly (choreography) or introduce a conductor (orchestration). After building several order platforms, I've gravitated toward orchestration for one critical reason: **operational visibility**.

### Choreography: The Peer-to-Peer Approach

With choreography, each service listens for events and reacts:

```
┌─────────┐    OrderPlaced    ┌──────────┐
│  Order  │ ────────────────▶│Payment │
└─────────┘                  └────────┘
                                    │
                           PaymentAuthorized
                                    ▼
                            ┌────────────┐
                            │ Inventory │
                            └────────────┘
```

**Pros:**
- Services remain loosely coupled
- No single point of failure
- Easy to add new participants

**Cons:**
- Hard to trace transaction flow
- Difficult to implement partial rollbacks
- Circular dependencies trap teams

### Orchestration: The Conductor Pattern

With orchestration, a central Saga Orchestrator coordinates the workflow:

```
┌─────────────┐
│   Saga     │◀────────────────────┐
│ Orchestrator                   │
└─────────────┘                   │
       │                         │
       ▼                         │
┌────────────┐    ReserveInventory
│    Order   │──────────────────▶│
└────────────┘                   │ InventoryReserved
       │                         ▼
       │                  ┌────────────┐
       │                  │   Payment  │
       │                  └────────────┘
       │                         │
       │                         ▼
       │                  ┌────────────┐
       │                  │  Shipping  │
       ▼                  └────────���───┘
```

**Our choice: orchestration** simplifies compensation logic, centralizes failure handling, and provides a single source of truth for order state. For a retail platform handling 10,000+ orders daily, this visibility is non-negotiable.

### Compensating Actions

When failures occur, the orchestrator executes compensating transactions:

| Step | Failure Point | Compensating Action |
|------|-------------|-------------------|
| 1. Inventory | Reserve fails | **ReleaseInventory** - Release reserved stock |
| 2. Payment | Authorization fails | **RefundPayment** - Reverse payment, release hold |
| 3. Shipping | Scheduling fails | **CancelShipping** - Cancel shipment booking |
| 4. All | Saga timeout | **CompensateAll** - Reverse in reverse order |

```typescript
// Compensating transaction handler
class OrderSagaOrchestrator {
  async compensate(step: SagaStep, context: SagaContext): Promise<void> {
    switch (step) {
      case 'INVENTORY_RESERVED':
        await this.releaseInventory(context.orderId);
        break;
      case 'PAYMENT_AUTHORIZED':
        await this.refundPayment(context.paymentId);
        break;
      case 'SHIPPING_SCHEDULED':
        await this.cancelShipping(context.shipmentId);
        break;
    }
  }

  async onSagaFailed(context: SagaContext, failedStep: SagaStep): Promise<void> {
    const completedSteps = this.getCompletedSteps(context);
    for (const step of completedSteps.reverse()) {
      try {
        await this.compensate(step, context);
      } catch (error) {
        await this.moveToDeadLetter(context.orderId, error);
      }
    }
  }
}
```

## Idempotency, Deduplication, and At-Least-Once Delivery

In distributed systems, network failures are inevitable. Your system must handle duplicate commands gracefully.

### Idempotency Through Unique Command IDs

Every command carries a unique identifier. The event store rejects duplicates:

```typescript
interface CommandHandler {
  async execute(command: Command): Promise<Result> {
    const isDuplicate = await this.eventStore.isDuplicate(command.commandId);
    if (isDuplicate) {
      return { success: true, duplicate: true };
    }
    return this.processCommand(command);
  }
}

// Event store uniqueness constraint
class EventStore {
  async append(event: Event): Promise<void> {
    await this.db.query(
      `INSERT INTO events (stream_id, command_id, event_type, payload)
       VALUES ($1, $2, $3, $4)
       ON CONFLICT (command_id) DO NOTHING`,
      [event.streamId, event.commandId, event.type, event.payload]
    );
  }
}
```

### Deduplication via Event Store Indexing

```sql
-- Indexed for fast deduplication lookups
CREATE INDEX idx_command_id ON events(command_id);
CREATE UNIQUE INDEX idx_command_id_unique ON events(command_id) WHERE command_id IS NOT NULL;
```

### At-Least-Once Delivery with Retry Mechanisms

```typescript
interface MessageBroker {
  async publish(command: Command): Promise<void> {
    const maxRetries = 3;
    for (let attempt = 1; attempt <= maxRetries; attempt++) {
      try {
        await this.sender.send(command);
        return;
      } catch (error) {
        if (attempt === maxRetries) throw error;
        await this.delay(Math.pow(2, attempt) * 1000);  // Exponential backoff
      }
    }
  }

  async moveToDeadLetter(command: Command, error: Error): Promise<void> {
    await this.deadLetterQueue.publish({
      command,
      error: error.message,
      timestamp: new Date(),
      retryCount: 0
    });
  }
}
```

A real-world example: Amazon's order system uses this pattern to handle flash sales. During Prime Day, duplicate paymentProcessing commands are inevitable when customers double-click. Our approach ensures no double-charges while guaranteeing processing.

## Per-Tenant Isolation and Snapshotting Strategy

Multi-tenant platforms require strict data boundaries. Each tenant gets isolated event streams.

### Per-Tenant Event Streams

```typescript
class TenantEventStore {
  getStreamId(tenantId: string, aggregateId: string): string {
    return `${tenantId}:${aggregateId}`;
  }

  async append(event: Event): Promise<void> {
    const streamId = this.getStreamId(event.tenantId, event.aggregateId);
    await this.db.query(
      `INSERT INTO tenant_events (tenant_id, stream_id, event)
       VALUES ($1, $2, $3)`,
      [event.tenantId, streamId, event]
    );
  }
}
```

### Snapshotting Strategy

Rebuilding aggregates from events works for small datasets. At scale, we snapshot:

```typescript
class SnapshotManager {
  private readonly SNAPSHOT_INTERVAL = 100;

  async maybeSnapshot(aggregate: OrderAggregate): Promise<void> {
    if (aggregate.version % this.SNAPSHOT_INTERVAL === 0) {
      await this.snapshotStore.save({
        aggregateId: aggregate.id,
        tenantId: aggregate.tenantId,
        state: aggregate.state,
        version: aggregate.version,
        timestamp: new Date()
      });
    }
  }

  async rebuild(aggregateId: string, tenantId: string): Promise<OrderAggregate> {
    const snapshot = await this.snapshotStore.getLatest(aggregateId, tenantId);
    const events = await this.eventStore.getAfter(aggregateId, snapshot.version);
    return this.replay([snapshot, ...events]);
  }
}
```

This approach handles tenants with millions of orders while keeping rebuild times under 100ms.

### Chaos Engineering Scenarios

| Scenario | Simulation | Expected Outcome |
|----------|-------------|-------------------|
| Payment service timeout | Kill payment pod | Saga initiates compensation within 30s |
| Database lock | Inject lock contention | Dead letter queue captures stalled orders |
| Network partition | Block service communication | Retry with exponential backoff succeeds |
| Inventory overcommit | Reserve 200% stock | Saga fails, compensation releases all |

### Failure Injection Testing


## Conclusion

Building a resilient order platform demands explicit handling of distributed uncertainty. Our saga orchestration approach provides:

1. **Centralized coordination** for operational clarity
2. **Explicit compensating actions** for reliable rollbacks
3. **Idempotent commands** to prevent duplicates
4. **At-least-once delivery** for guaranteed processing
5. **Per-tenant isolation** for security and performance
6. **Snapshotting** for reasonable rebuild times
