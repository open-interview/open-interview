> **Understanding Context: A payment system uses PostgreSQL with MVCC. Table accounts(id, balance). Design a concurrent transfer workflow: transfer amount from source account to destination account atomically, ensuring balance never goes negative, no money is lost or duplicated, and invariants hold under high contention. Provide concrete SQL steps, locking strategy (SELECT FOR UPDATE vs SERIALIZABLE), and a simple retry policy. What tests would you implement to prove correctness under parallel transfers?**

## Overview

Use an explicit transfer transaction with row-level locking to prevent race conditions. Example: BEGIN; SELECT balance FROM accounts WHERE id=$src FOR UPDATE; SELECT balance FROM accounts WHERE id=$dst FOR UPDATE; IF balance_src < $amount THEN ROLLBACK; ELSE UPDATE accounts SET balance = balance - $amount WHERE id = $src; UPDATE accounts SET balance = balance + $amount WHERE id = $dst; COMMIT; END IF; For higher contention, consider SERIALIZABLE isolation with retry logic.
