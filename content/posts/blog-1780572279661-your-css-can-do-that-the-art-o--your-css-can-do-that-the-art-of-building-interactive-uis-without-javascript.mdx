## Hook — The JavaScript Default Trap

Every developer has been there. A designer drops a new interaction pattern into a mockup — tabs, accordions, tooltips — and your hand instinctively reaches for React state or a jQuery plugin. It feels natural. It feels safe. But here is the uncomfortable question GitHub's team faced: what if your "safe" JavaScript is the very thing making your UI slow [1]? The Primer team discovered that their sticky sidebar pane, a seemingly simple UI element, was triggering expensive layout recalculations every time a user scrolled. The fix was not optimizing the JavaScript — it was cutting it out entirely. And that is when the real question emerges: how much of your UI actually needs JavaScript?

## Problem — The Hidden Cost of JavaScript Interactivity

JavaScript-based UI patterns come with invisible baggage. Every component that listens to scroll, resize, or click events adds computational overhead. React components re-render, style recalculations cascade, and the browser's layout engine works overtime [4]. For a simple tab panel — a UI pattern that has existed since the dawn of web applications — the typical React implementation involves state management, event handlers, and conditional rendering. On mobile, where every millisecond of scroll jank is magnified, this overhead becomes a user experience disaster. The problem is not that JavaScript cannot handle tabs. The problem is that you are paying a performance tax for something CSS has been able to do since 2013 [2].

## Real-World Case — GitHub's Primer Design System

GitHub's Primer design system team was building PageLayout, a core documentation page component used across all Primer-based documentation [1]. The sticky sidebar pane relied on a JavaScript hook (`useStickyPaneHeight`) that calculated the pane height on every scroll and resize event. This approach caused severe performance degradation: 60ms+ of style recalculations on every interaction, unnecessary React re-renders from the height calculation hook, and a bloated JavaScript bundle from the styled-components dependency. The team's solution was to replace the JavaScript height calculation with a CSS Grid-based layout that naturally handled sticky positioning without any runtime computation. The impact was dramatic — eliminated style recalculations, removed React re-render overhead, reduced bundle size, and delivered smooth scrolling across all Primer documentation pages. The lesson? Sometimes the most impactful performance optimization is removing code entirely.

## Deep Dive — The CSS-Only Interactive Pattern Playbook

CSS has evolved far beyond simple styling. The radio-button tab pattern exploits a fundamental browser behavior: the `:checked` pseudo-class [2]. When you pair radio inputs (with a shared `name` attribute for single-select behavior) with the adjacent sibling combinator (`~`), you get a fully functional tab switching mechanism. Here is how it works: each radio input controls visibility of its associated panel. When checked, its panel displays via `display: grid`; when unchecked, the panel hides via `display: none` [4]. This is not a hack — it is a well-documented CSS pattern that has been production-tested across major design systems. The pattern gets even more powerful with responsive design [9]. On desktop, panels render as a 2×2 grid using `grid-template-columns: repeat(2, 1fr)`. On mobile (under 768px), a single column layout with `grid-template-columns: 1fr` provides optimal readability. The `aspect-ratio: 16/9` property ensures consistent image containers across screen sizes [5]. Accessibility is another area where CSS-only patterns shine. The `:focus-visible` pseudo-class [7] ensures keyboard users see focus indicators only when navigating via keyboard — not after every mouse click. This is the kind of accessibility detail that JavaScript implementations often get wrong. Then there is motion. The `prefers-reduced-motion` media query [6] becomes your gatekeeper for entrance animations. Cards can fade and slide in with staggered animation delays, but only for users who have not opted out of motion. This respects user preferences while still providing delightful micro-interactions for those who want them.

These serve as your clickable tab headers. **Step 3:** Wrap each tab's content in a ``. **Step 4:** Use the `:checked ~ .panel` CSS selector to show only the active panel. **Step 5:** Apply grid layouts with responsive breakpoints. **Step 6:** Add `aspect-ratio: 16/9` to card image containers. **Step 7:** Layer on entrance animations gated by `prefers-reduced-motion`. **Step 8:** Style `:focus-visible` outlines to maintain keyboard accessibility.

## Code Example — A Production-Ready CSS-Only Tab Panel

Here is the complete implementation, building everything covered so far into a single, production-ready component:

```css
/* Tab container - radio inputs control panel visibility */
.tabs input:not(:checked) ~ .panel {
  display: none;
}

/* Panel becomes a responsive grid when active */
.tabs .panel {
  display: grid;
  grid-template-columns: repeat(2, 1fr);
  gap: 1.5rem;
}

/* Mobile: single column for readability */
@media (max-width: 768px) {
  .tabs .panel {
    grid-template-columns: 1fr;
  }
}

/* Fixed 16:9 image area for every card */
.card__image {
  aspect-ratio: 16 / 9;
  background: #eee;
  border-radius: 8px;
}

/* Entrance animation: only for users who want motion */
@media (prefers-reduced-motion: no-preference) {
  .card {
    animation: fadeSlideIn 0.3s ease-out backwards;
  }
  .card:nth-child(2) { animation-delay: 0.1s; }
  .card:nth-child(3) { animation-delay: 0.2s; }
  .card:nth-child(4) { animation-delay: 0.3s; }
}

@keyframes fadeSlideIn {
  from { opacity: 0; transform: translateY(10px); }
  to   { opacity: 1; transform: translateY(0); }
}

/* Keyboard-friendly focus indicators */
:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 2px;
}
```

The HTML structure pairs each radio input with a label for the tab header. The `~` adjacent sibling combinator connects the checked state to the panel visibility. Cards use `aspect-ratio` for consistent image containers, and the grid handles layout at both breakpoints. The staggering animation delay (0.1s increments) creates a cascading reveal effect that feels polished without being distracting [8].

## Lessons Learned — When to Go JavaScript-Free

Here is the insight that separates great frontend engineers from good ones: before reaching for a state management solution, ask whether CSS can solve the problem. The radio-button tab pattern proves that CSS is capable of sophisticated interactive behavior — tab switching, accordion panels, image carousels, and even modal dialogs can all be built without JavaScript [2][7]. The key criteria for going CSS-only are: the interaction involves binary visibility toggling (show/hide), the state is simple (checked/unchecked), and there is no need for dynamic data fetching or complex side effects. When these conditions are met, CSS-only patterns deliver superior performance, smaller bundle sizes, and better accessibility out of the box. GitHub's Primer team proved that eliminating JavaScript from sticky positioning made their documentation pages dramatically faster [1]. The same principle applies to tab panels, accordions, and countless other UI patterns. Your users will not notice the missing JavaScript. But they will definitely notice the snappy scrolling.

## Conclusion

The next time you build a tab panel, an accordion, or a content switcher, pause before writing that state variable. Ask yourself: can CSS handle this? GitHub's Primer team proved that eliminating JavaScript from interactive UI patterns is not just possible — it is often the superior choice. Your users will thank you with every smooth scroll, every instant tab switch, and every lightweight page load. Less JavaScript is not just a performance optimization. It is a design philosophy.
