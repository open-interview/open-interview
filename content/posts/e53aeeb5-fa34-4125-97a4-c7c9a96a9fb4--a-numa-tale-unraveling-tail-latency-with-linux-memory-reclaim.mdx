> **Picture this:** In 2013, LinkedIn faced intermittent tail latency spikes on NUMA servers during peak ingestion for an online-graph workload. Standard metrics looked healthy, yet tail latency persisted, traced to how Linux memory reclaim interacts with NUMA locality [1]. This journey walks through a concrete, real-world diagnostic workflow using only default Linux tools to confirm reclaim pressure, identify the NUMA node(s) and processes responsible, and apply safe mitigations that don’t disturb other tenants.

## Context: The problem that refuses to quit

Many developers confront the paradox: a system appears healthy in aggregate, but tail latency stubbornly lingers. Building on LinkedIn's experience that zone reclaim on NUMA systems can dominate tail latency [1], the goal is to verify whether kernel memory reclaim is pulling the tail, and if so, where the pressure is originating. The first step is to observe memory activity in real time and spot signs that reclaim is competing with app memory. The following command sequences establish baseline visibility and hint at reclamation pressure.

```
vmstat 1 20
```

Interpretation: look for si/so (swap in/out) spikes and cross-check with pgsteal trends in /proc/vmstat. If si/so spike while free memory remains seemingly adequate, reclamation pressure may be at fault. Example observation (abbreviated):

```
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
r  b   swpd   free   cache   buff  si so    bi  bo   in   cs us sy id wa st
 1  0      0  1024M  200M   50M  120  95  2  46   420  380  3  1 96  0  0
```

Additionally, inspect kernel memory reclaim indicators directly:

```
cat /proc/vmstat | grep -E 'pgsteal|pgfault|pgmajfault|pginod|pgscan'
```

If pgsteal increments align with si/so bursts, reclaim pressure is real rather than a transient anomaly. Inline citation [1] anchors this reasoning to the LinkedIn case study. 


## Journey: Mapping memory and NUMA locality

Building on the initial signal, the next discovery path maps memory across NUMA nodes and surfaces where reclamation pressure concentrates. Start with a top-down view of NUMA topology and per-node memory distribution. This reveals constrained nodes and potential hotspots that reclamation may be exploiting.

```
numactl --hardware
```

And inspect per-node memory statistics directly:

```
for n in /sys/devices/system/node/node*/meminfo; do echo "=== $n ==="; cat $n; done
```

These outputs identify how memory is allocated across nodes and whether some nodes are under memory pressure. Understanding current NUMA binding policies helps predict how memory pressure may cascade across tenants. Verify the current NUMA binding policy:

```
numactl --show
```

Finally, list memory-heavy processes to understand who might be driving demand:

```
ps aux --sort=-rss | head -20
```

Together, these steps reveal whether tail latency correlates with NUMA node pressure and which processes are the loudest memory consumers, setting up targeted mitigations. Transition: with the NUMA map in hand, the next move is to tailor mitigation without broad disruption. 


## Twist: The art of safe tuning without collateral damage

A surprising twist often emerges: the urge to blunt memory pressure with aggressive defaults can swing the balance against other tenants, especially in multi-tenant environments. The counterintuitive insight is that safety comes from targeted, conservative changes rather than sweeping rewrites. The story shifts to three precise mitigations that address reclaim pressure while preserving fairness and predictability.

1) Increase vm.min_free_kbytes conservatively (roughly 1% of total RAM). This helps keep pages readily reclaimable without triggering excessive paging under peak load.

```
TOTAL_KB=$(grep MemTotal /proc/meminfo | awk '{print $2}')
MIN_FREE_KB=$((TOTAL_KB / 100))
sudo sysctl -w vm.min_free_kbytes=${MIN_FREE_KB}
```

2) Tune vm.vfs_cache_pressure to 200 to favor reclaim of inode/dentry caches when memory is tight, reducing cache-related stalls:

```
sudo sysctl -w vm.vfs_cache_pressure=200
```

3) Pin critical ingestion/processing processes to isolated NUMA nodes using either numactl or taskset, so that important workloads don’t compete with less critical tenants for memory.

```
# Example: bind a critical PID 1234 to NUMA node 0
sudo numactl --cpunodebind=0 --membind=0 -p 1234
# Or launch a critical process on a specific node
sudo numactl --cpunodebind=0 --membind=0 /path/to/critical-service --args
```

These steps are deliberately cautious, designed to avoid regressing other workloads. After applying changes, verify observability to ensure performance improves without regressions. 


## Verification: Observability after changes

Post-mitigation, the same observability surfaces should be monitored to confirm the intended effect. Re-run the baseline checks and compare:

```
vmstat 1 20
```

```
cat /proc/vmstat | grep -E 'pgsteal|pgfault|pgmajfault'
```

```
numactl --hardware
```
```
ps aux --sort=-rss | head -20
```

If the tail latency improved while aggregate metrics remained healthy, the mitigations achieved their goal without destabilizing other tenants. If performance did not improve or worsened for other workloads, revert the changes or re-balance pinning to less aggressive bindings. The discipline here is incremental change and close observation. The LinkedIn case study reinforces this approach: careful VM tuning and NUMA-aware process affinity can yield latency improvements without broad disruption [1].

## Conclusion: What to do tomorrow

 tail latency in NUMA-heavy, multi-tenant environments often hides in plain sight—behind kernel memory reclamation and node topology. The three-part workflow—confirm reclaim pressure, identify NUMA nodes and processes, and apply conservative, targeted mitigations—offers a pragmatic path forward. Start by verifying reclaim indicators with vmstat and /proc/vmstat, map NUMA topology, and then apply minimal, reversible tunings to vm.min_free_kbytes and vm.vfs_cache_pressure, while isolating critical processes. The discipline of observability remains the strongest ally: continuous monitoring should be the default, not an afterthought. A memorable takeaway to share with the team: small, deliberate VM tuning guided by NUMA awareness can unlock dramatic latency improvements without destabilizing multi-tenant workloads.
