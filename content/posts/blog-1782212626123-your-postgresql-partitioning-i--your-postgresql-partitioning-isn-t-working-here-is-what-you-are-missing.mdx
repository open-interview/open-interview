## Hook — Your Partitioned Table Is Lying to You

You added partitioning to your PostgreSQL table. Maybe you celebrated. "Problem solved," you thought. Then that query on a specific date range fired, and... the database yawned. Another sequential scan. Another 30-second timeout. Sound familiar? Partitioning is not magic — it is a tool that only works when you understand how the planner actually uses it. Many developers discover that partitioning alone is like organizing a library by floor but then refusing to use the elevator.

## Problem — When Partitions Become Paperweights

A 100-million-row table partitioned by date sounds like a solid plan. The problem is that partitioning is just the first layer of defense. If your queries still scan every partition, or worse — scan every row within a partition — you have not gained anything. The database still does the same amount of work. The three most common culprits? First, partition pruning that silently fails because of type mismatches or function wrappers around the partition key. Second, indexes that exist but never get used because the planner optimistically chooses a sequential scan. Third, sort operations on unindexed columns that turn a simple filter into a memory monster.

## Real-World Case — CoinGecko

CoinGecko ran into this wall at full speed. Their 1TB+ partitioned table storing crypto price data across 8 years was hitting daily IOPS limits across multiple AWS replicas. Replica lag was growing, their Apdex score was falling, and the team kept cranking up IOPS — 24K just to tread water. The root cause? Partition pruning was working at the table level, but within each partition, queries were scanning millions of rows because the indexes did not align with their filter patterns. The fix was deceptively simple: composite indexes that matched the actual WHERE clause pattern, plus strategic use of CLUSTER to physically reorder data. The results speak for themselves: p99 latency dropped 86% from 4.13 seconds to 578 milliseconds, IOPS demand fell 20%, and replica lag vanished [1]. Queries that previously timed out at 30+ seconds now returned in milliseconds.

## Deep Dive — Reading the EXPLAIN Tea Leaves

The EXPLAIN (ANALYZE, BUFFERS) output tells you a story — you just need to know how to read it. Here is what to look for. First, the "Subplans Removed" line: that is your partition pruning verification. If it says 0, your partition key might be wrapped in a function like DATE(event_date) instead of using the raw column, which silently defeats pruning. Second, the node type: "Seq Scan on events_p2024_01" means the planner decided a full scan was cheaper than using an index. That is a smell — either your query is too broad, or you need a composite index [2]. Third, "Sort Method: external merge" indicates your sort spilled to disk. Adding an index on the sort column eliminates this. Fourth, "Buffers: shared hit=120 read=4500" tells you how much of the data was in memory vs. read from disk. A high read count compared to hit means your working set does not fit in shared buffers, and better indexing can drastically reduce the total buffer access. The key insight: EXPLAIN is not a report card — it is a diagnostic. It tells you exactly which muscle needs exercise.

## Workflow — From Slow Query to Milliseconds in 5 Steps

Here is the workflow that CoinGecko's team followed, and that you can replicate today. Start by capturing the slow query with EXPLAIN (ANALYZE, BUFFERS, TIMING). Verify partition pruning by checking the "Subplans Removed" count. If it is unexpectedly low, check for type casts or function calls on the partition key. Inspect the node types: any Seq Scan inside a targeted partition is a red flag. Build a composite index that covers both the partition key and the most selective filter column — this lets the database navigate directly to the right rows instead of scanning. Finally, consider CLUSTER to physically reorder the table on disk to match the index order, which improves locality of reference and reduces the number of pages scanned. ## Code Example — Diagnosing and Fixing Partition Performance

The first step is always diagnosis. Run EXPLAIN with ANALYZE and BUFFERS to see what the planner is actually doing. The BUFFERS option is critical — it tells you how much data was read from disk versus served from memory, which is your real cost [3]. Once you confirm partition pruning is working but sequential scans remain, add a composite index that matches your filter pattern. The order matters: put the partition key first (enables partition pruning), then the most selective filter column. This lets PostgreSQL quickly locate the right partition and then use the second column to narrow down rows within it. After creating the index, consider CLUSTER to physically reorder the table. Note that CLUSTER is a blocking operation and needs an ACCESS EXCLUSIVE lock, so plan for maintenance windows or use pg_repack for zero-downtime reordering [4]. The results can be dramatic — what took a sequential scan of millions of rows now becomes an index-only scan touching a handful of pages.

## Lessons Learned — What CoinGecko's Story Teaches Us

If there is one takeaway, it is this: partitioning is a foundation, not a solution. You still need to verify that each partition is efficiently queryable. The greatest insight from CoinGecko's experience is that 86% latency reduction came not from throwing hardware at the problem, but from aligning the index structure with the actual query patterns [1]. Here are the rules to carry forward. Always confirm partition pruning with EXPLAIN — never assume it works. Build composite indexes that match your WHERE clause, not just single-column indexes on every column. Profile buffer usage, not just execution time — disk reads are the real cost. Use CLUSTER for read-heavy, range-scan-heavy tables, but plan for the lock. And finally, revisit your indexes after each schema change; what worked last quarter might not work today [5]. The database is not magic. It just needs you to ask the right questions.

## Conclusion

The next time someone tells you partitioning is the answer, nod and agree — then open an EXPLAIN output and check whether it is actually working. Partitioning plus aligned indexing plus physical ordering is the real recipe. CoinGecko cut their p99 from 4.13 seconds to 578 milliseconds doing exactly this. The fix is not more hardware. It is understanding what your database is actually doing and giving it the tools to do less work.
