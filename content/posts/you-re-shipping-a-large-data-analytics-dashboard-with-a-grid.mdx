> **Understanding You're shipping a large data analytics dashboard with a grid rendering 800+ columns; the initial bundle is 2.3MB and Lighthouse scores poorly due to heavy scripting. Propose a concrete approach to implement progressive hydration with per-column lazy-loading: how would you structure code-splitting, virtualization, and worker offloading, and how would you validate improvements using Lighthouse budgets and metrics?**

## Overview

Implement progressive hydration for the grid by splitting columns into visible and offscreen groups. Hydrate only visible columns on initial mount, then lazy-load remaining columns via dynamic imports triggered by scroll/viewport events. Leverage IntersectionObserver to detect column visibility and use dynamic imports for code-splitting. Apply virtualization to render only visible rows, significantly reducing DOM nodes. Offload data processing and complex calculations to Web Workers to maintain main thread responsiveness. Validate improvements using Lighthouse budgets, monitoring First Contentful Paint, Time to Interactive, and script execution time metrics.
