> **Understanding Scenario: In a cross-region ride-hailing platform, live trip events are cached in Redis for hot path, stored in MongoDB for rider/driver state, streamed to Cassandra as per-vehicle telemetry, and archived in DynamoDB for billing. For a new trip booking, design a concrete write path: specify concrete keys and document shapes, TTLs, and how data flows between stores when a ride starts, completes, and settlement occurs. Include idempotence, deduplication, ordering guarantees, and cross-store cache coherence?**

## Overview

The booking_id serves as the idempotent key. Redis stores the hot path data with key pattern trip:booking:{booking_id} and a 15-minute TTL for in-flight state. MongoDB contains the canonical trip state in the trips collection with documents structured as {booking_id, rider_id, driver_id, start_ts, status, updated_at}. Cassandra handles time-series telemetry in a table partitioned by vehicle_id with clustering by timestamp for ordered event storage. DynamoDB stores immutable billing records with {booking_id, rider_id, driver_id, fare_amount, payment_status, created_at} for audit trails.
