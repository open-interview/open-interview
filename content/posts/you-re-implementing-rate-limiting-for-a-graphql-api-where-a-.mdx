> **Understanding You're implementing rate limiting for a GraphQL API where a single query can request multiple nested resources, making traditional request-count based limiting ineffective. How would you design a rate limiting strategy that accounts for query complexity, depth, and resource cost? How would you handle expensive queries like { users { posts { comments { likes } } } } differently from simple queries?**

## Overview

Implement query cost analysis by assigning weighted costs to fields based on their computational overhead and query depth. Calculate the total cost for each request and apply rate limits against a user's cost budget rather than simple request counts. Additionally, enforce depth limiting to prevent exponentially expensive nested queries.
