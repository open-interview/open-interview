> **Understanding You're designing a global time-series store for multi-tenant SaaS. Data is written in 1-hour shards per tenant and must survive regional outages. Design a scalable sharding and replication strategy across 3 regions that isolates tenants, supports hot tenants, and controls storage costs. Compare fixed-size vs sliding-window shards, decide on replication factor, and outline a test plan for cross-region failover and data consistency?**

## Overview

Implement 1-hour rolling shards per tenant using a composite key (tenantId + timestamp). Replicate each shard across all three regions with quorum-based consistency (W=2, R=2) to ensure durability and read availability during regional outages.

**Fixed-size vs. sliding-window shards:** Use fixed-size shards (1-hour windows) for predictable storage allocation and easier capacity planning. Sliding-window shards offer finer granularity but introduce operational complexity. For hot tenants, implement dedicated shard pools with higher replication factors and provisioned throughput to prevent resource contention.

**Replication strategy:** Set replication factor of 3 (one per region) with regional affinity to minimize cross-region latency for reads. Use eventual consistency with configurable sync windows—hot tenant data can use synchronous cross-region replication, while cold data remains asynchronously replicated to reduce costs.

**Storage cost control:** Implement tiered storage with hot (SSD), warm (HDD), and cold (object storage) tiers based on data age. Enable TTL-based automatic purging and implement compression at the shard level.
