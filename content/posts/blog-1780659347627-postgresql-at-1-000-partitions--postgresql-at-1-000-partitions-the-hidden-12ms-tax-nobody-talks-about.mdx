## Hook — What If Your Database Is Spending 35x More Time Planning Than Executing?

Imagine this: you run an EXPLAIN ANALYZE on a query you have tuned to perfection. Execution time: 0.35ms. Planning time: 12.4ms. The database spent 35 times longer figuring out *how* to get your data than actually getting it. That is the reality PostgresAI engineers documented when they tested PostgreSQL 16 across 0 to 1,000 partitions [1]. And here is the plot twist that changes everything: subsequent queries in the *same database session* dropped planning time back below 0.1ms, regardless of partition count. The problem was never the partitions themselves — it was how PostgreSQL's planner warms up its cache for new connections.

## Problem — Partitions Are a Double-Edged Sword

Partitioning is one of the most powerful tools in a PostgreSQL DBA's arsenal. It enables time-based data retention, improves query performance through partition pruning, and makes vacuum manageable on multi-terabyte tables. But it comes with a hidden cost that many developers discover the hard way. The PostgreSQL planner must examine every partition's metadata, lock all indexes involved, and build a plan that accounts for the full inheritance hierarchy. This overhead scales with partition count — linearly, in fact [1]. The official documentation itself warns that planning times grow and memory consumption increases when more partitions survive pruning [2]. The danger zone typically starts around 1,000 partitions and becomes painful past 5,000 to 10,000. For OLTP workloads — web apps, mobile backends, real-time APIs — where every millisecond matters, this planning tax can quietly destroy your p99 latency.

## Real-World Case — PostgresAI

In October 2024, PostgresAI published the results of a meticulously designed experiment that quantified the relationship between partition count and planning time in PostgreSQL 16 [1]. The methodology was elegant: create a partitioned table with a primary key, run EXPLAIN (ANALYZE, BUFFERS) on a simple lookup query, then progressively add partitions from 0 to 1,000 — measuring the impact at each step. The results were striking. For the *first query in a session*, planning buffer hits grew from ~50 to ~950 — a 19x increase in catalog metadata reads. Planning time followed the same trajectory, climbing from ~0.1ms to ~12.4ms. But the most important finding came when they reran the experiment: subsequent queries in the same session showed planning times below 0.1ms consistently. Buffer hits dropped to just 3 (24 KiB) regardless of partition count. The implications are profound for anyone managing connection pools or dealing with connection-per-query patterns in serverless or lambda architectures.

## Deep Dive — What Actually Happens During Planning?

When PostgreSQL plans a query against a partitioned table, it performs several operations that many developers never see. First, the planner opens the relation and discovers all child partitions by scanning pg_inherits — this means more catalog lookups with every new partition. Second, it acquires AccessShareLock on every partition and all their indexes [1]. For 1,000 partitions with 2-3 indexes each, that is 3,000+ lock acquisitions before a single row is read. Third, it builds a partition descriptor in the relcache, which involves memory allocation proportional to the partition count. The PostgresAI experiment showed that the first query in a session bears the full brunt of this work because the relcache starts cold. This is why connection pooling makes such a dramatic difference — persistent connections keep the relcache warm. David Rowley, a PostgreSQL core contributor, confirmed this on social media, calling the initial results misleading because most production systems use connection pooling [1]. This leads to a counterintuitive insight: the planning time problem with many partitions is largely a *cache warming* problem, not a fundamental algorithmic limitation. However, this doesn't mean you can ignore partition counts entirely. More complex queries with multiple joins, subqueries, or window functions amplify the planning overhead because the planner must evaluate path options against every non-pruned partition.

## Workflow — Diagnosing and Fixing Slow Partition Queries

When you encounter a slow query on a partitioned table, follow this systematic diagnostic workflow. The Mermaid diagram below maps the decision tree you will traverse. Start by running EXPLAIN (ANALYZE, BUFFERS) on the problematic query. First, verify partition pruning is working — you should see "Subplans Removed" or only a subset of partitions mentioned. If all partitions appear, your partition key does not align with your WHERE clause. Next, check if the query uses Index Scan or Sequential Scan. A sequential scan on a large partition is your biggest red flag — it needs a composite index covering both the partition key and the filtered columns. Then examine sort and hash operations. Expensive sorts suggest missing indexes on ORDER BY columns. Finally, look at the planning time reported by EXPLAIN ANALYZE. If first-query planning time exceeds 5-10ms and you have hundreds of partitions, the solution is often connection pooling rather than redesign. Retest after each change to isolate the improvement.

## Lessons Learned — Five Principles for Partitioned Table Performance

After examining the PostgresAI experiment, the PostgreSQL documentation, and decades of production experience, five clear principles emerge. First, keep your partition count under 1,000 — ideally under 100. The official documentation recommends staying below "a few thousand" [2], and the PostgresAI data shows linear degradation starts immediately [1]. Second, use connection pooling. PgBouncer or similar tools keep relcache entries warm, eliminating the first-query penalty that dominated the PostgresAI results. Third, monitor planning time explicitly. Enable pg_stat_statements.track_planning in postgresql.conf [6] and track mean_plan_time per query — this surfaces issues before users complain. Fourth, create composite indexes that cover both the partition key and your most common filter columns. A single index on (event_date, status) can turn a Sequential Scan into an Index Scan. Fifth, consider CLUSTER for tables where date-range scans are the primary access pattern — physically ordering data on disk can reduce the number of pages read by an order of magnitude [4].

## Conclusion

The 12ms planning tax at 1,000 partitions is real, but it is also solvable. The PostgresAI experiment taught us something deeper than a simple performance warning: it revealed how PostgreSQL's planner behaves under load and why connection management is just as important as query optimization. The next time you add a partition to your table, ask yourself three questions: Are you monitoring planning time or just execution time? Is your connection pooler configured to keep sessions alive? And does your index strategy match how you actually query the data? The difference between a database that feels sluggish and one that flies is often not about hardware — it is about understanding the hidden work that happens before a single row is fetched.
