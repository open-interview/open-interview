> "In distributed systems, failure isn't a question of if—it's a question of when. The difference between a brittle system and a resilient one lies in how you orchestrate recovery."

Imagine this: A media company uploads 10,000 articles for processing. Your pipeline encodes videos, generates thumbnails, and publishes to CDN. Then CDN publish fails. Do you have a plan to roll back cleanly, charge correctly, and maintain auditability? If not, you're about to learn.

## The Challenge: Multi-Tenant Content Pipeline at Scale

Every day, platforms like Medium, YouTube, and news outlets process millions of articles and videos. Each tenant (organization or user) expects:
- **Isolation**: Their content never touches another tenant's resources
- **Exactly-once processing**: No duplicate encoding, no missed billing events
- **Consistent failures**: If step 3 fails, steps 1-2 roll back completely

Traditional approaches using synchronous calls and database transactions break down when steps span services, regions, and billing systems.

## Architecture: CQRS + Event Sourcing + Saga

We combine three powerful patterns:

1. **CQRS (Command Query Responsibility Segregation)**: Separate read and write models for optimal performance
2. **Event Sourcing**: Store state changes as an immutable event log—essential for auditability and replay
3. **Saga Orchestrator**: Coordinate distributed steps with compensating actions for rollback

### The Event Flow

```
[Upload] → [Encoding] → [Thumbnails] → [CDN Publish]
              ↓              ↓              ↓
         StepState(1)  StepState(2)  StepState(3)
              ↓              ↓              ↓
         BillEvent(1)  BillEvent(2)  BillEvent(3)
```

Every event carries `tenantId` and `articleId`. This enables:
- Per-tenant outbox processing
- Isolation via tenant-scoped storage partitions
- Complete audit trail

## Implementation: The Saga Orchestrator

Here's a minimal sketch showing the orchestrator starting a saga and handling compensation:

```typescript
// types.ts
interface PipelineEvent {
  tenantId: string;
  articleId: string;
  step: 'encoding' | 'thumbnails' | 'cdn_publish';
  status: 'started' | 'completed' | 'failed' | 'compensated';
  timestamp: Date;
  payload: Record<string, unknown>;
}

interface StepState {
  tenantId: string;
  articleId: string;
  step: string;
  status: 'pending' | 'in_progress' | 'completed' | 'failed' | 'compensated';
  idempotencyKey: string;
  compensatingAction?: string;
  log: string[];
}

// saga-orchestrator.ts
class ContentPipelineSaga {
  private readonly stepStore: StepStore;
  private readonly eventStore: EventStore;
  private readonly billingService: BillingService;
  private readonly cdnClient: CDNClient;

  async startSaga(tenantId: string, articleId: string, contentUrl: string): Promise<string> {
    const sagaId = `saga-${tenantId}-${articleId}-${Date.now()}`;
    
    await this.eventStore.append({
      tenantId,
      articleId,
      type: 'SagaStarted',
      payload: { sagaId, contentUrl },
    });

    // Begin the orchestration
    await this.executeStep(sagaId, tenantId, articleId, 'encoding', async () => {
      return await this.runEncoding(tenantId, articleId, contentUrl);
    });

    return sagaId;
  }

  private async executeStep(
    sagaId: string,
    tenantId: string,
    articleId: string,
    step: string,
    action: () => Promise<void>
  ): Promise<void> {
    const idempotencyKey = `${tenantId}:${articleId}:${step}`;
    
    // Check for exactly-once using StepState
    const existingState = await this.stepStore.get(idempotencyKey);
    if (existingState?.status === 'completed') {
      console.log(`Step ${step} already completed, skipping`);
      return;
    }

    // Record in-progress state
    await this.stepStore.save({
      tenantId,
      articleId,
      step,
      status: 'in_progress',
      idempotencyKey,
      log: [`Started ${step} at ${new Date().toISOString()}`],
    });

    try {
      await action();
      
      // Mark step complete
      await this.stepStore.updateStatus(idempotencyKey, 'completed');
      
      await this.eventStore.append({
        tenantId,
        articleId,
        type: `StepCompleted`,
        payload: { sagaId, step },
      });

      // Emit billing event
      await this.billingService.emitStepEvent(tenantId, articleId, step, 'completed');

    } catch (error) {
      await this.stepStore.updateStatus(idempotencyKey, 'failed');
      throw error;
    }
  }

  async handleFailure(
    sagaId: string,
    tenantId: string,
    articleId: string,
    failedStep: string,
    error: Error
  ): Promise<void> {
    console.error(`Saga ${sagaId} failed at ${failedStep}: ${error.message}`);
    
    // Trigger compensating actions in reverse order
    const completedSteps = await this.getCompletedSteps(tenantId, articleId);
    const stepsToCompensate = completedSteps.reverse();

    const compensationLog: string[] = [];

    for (const step of stepsToCompensate) {
      try {
        await this.compensateStep(tenantId, articleId, step);
        compensationLog.push(`Compensated ${step} successfully`);
      } catch (compError) {
        compensationLog.push(`Compensation failed for ${step}: ${compError.message}`);
        // Log but continue with other compensations
      }
    }

    // Emit PipelineFailed event with full audit log
    await this.eventStore.append({
      tenantId,
      articleId,
      type: 'PipelineFailed',
      payload: {
        sagaId,
        failedStep,
        error: error.message,
        compensationLog,
        timestamp: new Date().toISOString(),
      },
    });

    // Emit failure billing event
    await this.billingService.emitStepEvent(tenantId, articleId, failedStep, 'failed');
  }

  private async compensateStep(tenantId: string, articleId: string, step: string): Promise<void> {
    const compensationActions: Record<string, () => Promise<void>> = {
      cdn_publish: async () => {
        await this.cdnClient.delete(`${tenantId}/articles/${articleId}`);
        console.log(`Rolled back CDN publish for ${tenantId}/${articleId}`);
      },
      thumbnails: async () => {
        await this.thumbnailService.deleteThumbnails(tenantId, articleId);
        console.log(`Deleted thumbnails for ${tenantId}/${articleId}`);
      },
      encoding: async () => {
        await this.encodingService.deleteAsset(tenantId, articleId);
        console.log(`Deleted encoded asset for ${tenantId}/${articleId}`);
      },
    };

    const action = compensationActions[step];
    if (!action) {
      throw new Error(`No compensation action defined for step: ${step}`);
    }

    await action();

    await this.stepStore.updateStatus(
      `${tenantId}:${articleId}:${step}`,
      'compensated'
    );
  }
}
```

## Per-Tenant Isolation: The Key to Multi-Tenancy

Isolation isn't just about security—it's about ensuring one tenant's failure doesn't cascade to others:

```typescript
// tenant-isolation.ts
class TenantScopedStore {
  private getTenantPartition(tenantId: string): string {
    return `tenant_${tenantId}`;
  }

  async saveStepState(state: StepState): Promise<void> {
    const partition = this.getTenantPartition(state.tenantId);
    await this.store.put(`${partition}/steps/${state.idempotencyKey}`, state);
  }

  async getStepState(tenantId: string, idempotencyKey: string): Promise<StepState | null> {
    const partition = this.getTenantPartition(tenantId);
    return await this.store.get(`${partition}/steps/${idempotencyKey}`);
  }

  async appendEvent(event: PipelineEvent): Promise<void> {
    const partition = this.getTenantPartition(event.tenantId);
    await this.eventStore.append(`${partition}/events/${event.articleId}`, event);
  }
}
```

This pattern ensures:
- Tenant data never shares storage partitions
- Queries are scoped to single tenant (no cross-tenant leaks)
- Billing events are always attributed correctly

## Exactly-Once Processing: The Idempotency Key

The critical piece for exactly-once is the `StepState` keyed by `(tenantId, articleId, step)`:

```typescript
// idempotency-handler.ts
class IdempotencyHandler {
  async processIfIdempotent<T>(
    tenantId: string,
    articleId: string,
    step: string,
    handler: () => Promise<T>
  ): Promise<T> {
    const key = `${tenantId}:${articleId}:${step}`;
    const existing = await this.stepStore.get(key);

    // Already completed - return cached result
    if (existing?.status === 'completed') {
      return existing.result as T;
    }

    // Failed or in-progress - allow retry
    if (existing?.status === 'failed' || existing?.status === 'in_progress') {
      return await handler();
    }

    // First attempt - execute and record
    const result = await handler();
    await this.stepStore.save({
      key,
      status: 'completed',
      result,
      idempotencyKey: key,
    });

    return result;
  }
}
```

## Actionable Conclusion

Building a resilient multi-tenant content pipeline requires:

1. **Tag every event** with `tenantId` and `articleId` for isolation and audit
2. **Use StepState** keyed by `(tenantId, articleId, step)` for exactly-once semantics
3. **Implement compensating actions** in reverse order with full logging
4. **Emit billing events** per step completion/failure for accurate tenant charges
5. **Store events immutably** for replay capability and debugging

The saga orchestrator pattern transforms what could be a cascade of failures into a manageable, auditable, and billable workflow. Start with the minimal sketch above, adapt to your infrastructure, and sleep better knowing your pipeline can handle failure gracefully.

Remember: In distributed systems, the question isn't *if* something will fail—it's *when* your system will recover elegantly.
