> **Understanding You have a microservices-based e-commerce order processing system where creating an order involves reserving inventory, processing payment, and scheduling delivery. If payment fails after inventory has been reserved, how would you design a compensation strategy to ensure data consistency across services? Compare choreography-based vs orchestration-based Saga patterns and discuss how you would handle partial failures and implement idempotency to prevent duplicate operations.**

## Overview

I would implement the Saga pattern using an orchestration-based approach for this distributed transaction scenario. The central Order Service acts as the orchestrator, sequentially invoking the Inventory, Payment, and Delivery services while managing all compensation logic. When payment fails after inventory has been reserved, the orchestrator triggers compensating transactions to release the reserved inventory and reverse any associated loyalty points. I would also ensure idempotency through unique request IDs and idempotency keys at each service level to prevent duplicate operations from network retries.
