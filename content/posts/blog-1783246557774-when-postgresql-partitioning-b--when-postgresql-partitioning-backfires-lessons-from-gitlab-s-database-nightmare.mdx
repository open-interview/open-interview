## Hook — The Partition That Broke the Pipeline

You have done everything right. You read the PostgreSQL docs cover to cover. You designed a careful partitioning scheme by date. You migrated 100 million rows. Yet when you query a specific date range, the response time is... worse? Welcome to a club that GitLab never wanted to join. After rolling out a massive partitioning effort across their CI/CD database, GitLab discovered that some of their most critical queries were silently scanning every single partition—even when the data was neatly organized by date. This is the story of how partitioning nearly broke one of the largest PostgreSQL deployments on the planet, and what you can learn from it.

## Problem — Why Partitioning Can Make Things Worse

Partitioning is like organizing a library by floor. You put all 2023 books on the first floor, 2024 on the second, and so on. When a patron asks for a book from 2024, you send them directly to that floor. This is partition pruning, and it is the whole point of partitioning. But here is the problem: if the patron just asks for 'books' without specifying the year, the librarian has to check every single floor. In PostgreSQL terms, that means a full scan of every partition—and with each new partition you add, the query gets slower. Many developers assume that partitioning is a magic bullet. The truth is more nuanced. Partitioning optimizes for a specific query pattern: filtering by the partition key. If your application queries don't consistently include that key, you are not just failing to optimize—you are actively making things worse. Each new partition adds overhead to sequential scans, index maintenance, and query planning. The performance cliff is real, and it hits when you least expect it.

## Real-World Case — GitLab's Partitioning Meltdown

GitLab learned this lesson the hard way. Their CI/CD tables—specifically `ci_builds` and `ci_pipelines`—had grown to terabyte scale. After a multi-year engineering effort, they rolled out PostgreSQL declarative partitioning across these tables. The results looked good—until they weren't. Two Severity 2 incidents (INC-8367 and INC-8474) exposed a critical flaw. High-frequency Sidekiq workers and API authentication queries were querying the partitioned tables without including the `partition_id` filter. As GitLab created more partitions over time, these queries performed progressively worse, scanning every partition instead of one. The impact was severe. Database lock contention spiked as hundreds of thousands of LockManager slowpath lock acquisitions saturated replicas per second. Sidekiq queues backed up, violating SLOs. Customers experienced slow CI job status updates. The team had to halt partition creation entirely until they could trace and fix every query path. Post-fix, LockManager contention dropped dramatically. Partition creation was re-enabled with confidence. CI replica stability was restored [1]. The lesson? Partitioning is only as effective as the queries that use it.

## Deep Dive — Reading the EXPLAIN Plan Like a Detective

When a query on a partitioned table is slow, your first stop is the EXPLAIN plan. Think of it as a detective's notebook—every clue is there if you know where to look. First, check partition pruning. Look for `Subplans Removed` in the EXPLAIN output with `(ANALYZE, BUFFERS)`. PostgreSQL 12+ will tell you exactly how many partitions it pruned versus scanned. If you see all partitions being scanned, you have found the root cause [3]. Second, check index utilization. Is the query using an index or performing a sequential scan across partitions? A sequential scan on a partitioned table is especially dangerous because it compounds across every partition. Third, look for expensive sort or hash aggregate operations. PostgreSQL may choose to sort in memory or on disk, and with partitioned data, the cost estimation can be misleading [6]. The fix often involves creating composite indexes that include both the partition key and the columns in your WHERE clause. For example, if you partition by `event_date` and frequently filter by `status`, create an index on `(event_date, status)`. This satisfies both the partition pruning requirement and the filter clause [4]. You might also consider `CLUSTER` to physically reorder the table data to match the index, which can dramatically reduce random I/O. But be warned: CLUSTER is a one-time operation that locks the table and cannot be run concurrently. Use it judiciously [5].

## Workflow — The Step-by-Step Diagnostic Protocol

When you encounter a slow query on a partitioned table, follow this diagnostic workflow: Start by running `EXPLAIN (ANALYZE, BUFFERS)` on your query. Check if partition pruning is working by looking at the number of partitions scanned versus removed. If pruning is not working, the most likely cause is the query missing the partition key filter—add it. If pruning is working but the query is still slow, check what scan type is being used. A sequential scan on a single partition might be acceptable for small partitions, but for large ones you need an index. Create a composite index on `(partition_key, filtered_column)`. If an index is used but there is an expensive sort, consider whether `CLUSTER` or a covering index could help [2]. Re-test with EXPLAIN ANALYZE and iterate until the performance meets your SLOs. ## Code Example — Diagnosing and Fixing Slow Partitioned Queries

The first step is always diagnosis. Run EXPLAIN ANALYZE with the BUFFERS option to see exactly what PostgreSQL is doing. The output reveals whether partition pruning is working and what type of scan is being performed. If you see sequential scans across many partitions, you need an index. Create a composite index CONCURRENTLY to avoid blocking writes in production. Finally, consider CLUSTER to physically reorder the table data for optimal range scan performance.

## Lessons Learned — What Every Developer Should Take Away

Partitioning is not a set-and-forget optimization. It requires ongoing vigilance. First, always include the partition key in your WHERE clause—this is the single most important rule. Second, monitor your EXPLAIN plans as partitions are added. What works with 10 partitions may not work with 100 [7]. Third, invest in composite indexes pairing the partition key with your most common filter columns. Fourth, use CLUSTER carefully—it is powerful but expensive. Fifth, build observability into your database layer. GitLab's ability to trace the issue to specific Sidekiq workers was only possible because they had detailed monitoring [1]. Finally, make EXPLAIN plans part of your code review process. A query that works on a development database with 10,000 rows may fail spectacularly on production with 100 million rows [8]. Start by running EXPLAIN on your slowest partitioned query today.

## Conclusion

Partitioning is a powerful tool, but it demands respect. GitLab's story is a reminder that every architectural decision comes with hidden assumptions — and those assumptions only reveal themselves at scale. The next time you design a partitioned table, ask yourself: what queries will touch this data, and do they all include the partition key? Your future self, debugging a Sev-2 incident at 3am, will thank you. Start by running EXPLAIN on your slowest partitioned query today. You might be surprised at what you find.
