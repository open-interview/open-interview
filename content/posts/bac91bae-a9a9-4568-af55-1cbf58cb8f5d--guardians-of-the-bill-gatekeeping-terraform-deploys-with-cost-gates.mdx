## Architecture: Minimal Backend & Wrapper Module

A pragmatic starting point uses a single backend with per-env isolation and a wrapper module to route environment context to provider aliases. The backend keeps state for both dev and prod, while the wrapper module lets resources select the correct provider alias at plan time. This approach preserves simplicity while enabling accurate cost breakdowns per environment. 

Backend: a minimal S3 backend with DynamoDB lock (dev and prod share the same bucket, but use distinct keys):

```hcl
terraform {
  backend "s3" {
    bucket         = "tf-state-bucket"
    key            = "envs/dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "tf-lock"
  }
}
```

Wrapper module (wire env to provider aliases):

```hcl
# modules/wrap-env/variables.tf
variable "environment" { type = string }
variable "region" { type = string }

# modules/wrap-env/providers.tf
provider "aws" {
  alias  = "dev"
  region = var.region
}
provider "aws" {
  alias  = "prod"
  region = var.region
}

# modules/wrap-env/main.tf
locals {
  tf_provider = var.environment == "prod" ? aws.prod : aws.dev
}

# Example usage inside a root module:
# module "service" {
#   source = "./modules/service"
#   providers = { aws = local.tf_provider }
#   environment = var.environment
#}
```

References for these patterns come from the Terraform ecosystem and AWS provider guidance, illustrating how per-env configuration can live under a single backend while remaining isolated per environment [2][3].

## CI Gate: Plan, Export, Infracost, and Enforce Budget

The gating logic lives in the CI pipeline. A pull request triggers a plan, exports plan.json, runs an infracost breakdown, and fails if the monthly estimate exceeds a defined budget. This gives engineers actionable cost guidance at the exact moment of change and prevents drift from slipping into production.

GitHub Actions sketch (cost gate gatekeeping):

```yaml
name: Plan with Cost Gate
on: [pull_request]
jobs:
  plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v1
      - run: terraform init
      - run: terraform plan -out=tfplan
      - run: terraform show -json tfplan > plan.json
      - run: infracost breakdown --path plan.json
      - run: |
          BUDGET=100
          COST=$(infracost breakdown --path plan.json --format json | jq -r .timeframe[0].cost)
          if (( $(echo "$COST" > "$BUDGET" | bc -l) )); then exit 1; fi
```

Why this works: plan.json provides a snapshot of what would be created, infracost translates that into a dollar estimate, and the gate enforces the budget before any apply can proceed. This aligns with Bourne Leisure’s real-world approach of catching cost early in CI [1], while leveraging the standard Terraform workflow of plan before apply [4].

## Putting It All Together: When Would You Use This?

This pattern shines in multi-environment deployments where a single backend manages state but per-env isolation is essential. It’s particularly valuable when teams want:

- Early visibility into potential monthly spend per change [1]
- A scalable gate that can handle dev and prod with a single backend [2][3]
- A wrapper module that cleanly wires environment context to provider aliases [2]
- A repeatable CI gate that prevents costly surprises in production [4]

Hindsight suggests a few battle scars to watch:
- Avoid over-complicating provider alias wiring; keep the wrapper minimal and explicit.
- Ensure plan.json is generated consistently across environments to avoid mismatch in infracost input.
- Treat the budget as a living target; update it as teams learn from real deployments. 

Real-world proof anchors these lessons in practice and demonstrates how cost governance can become a natural, repeatable part of the workflow [1].
