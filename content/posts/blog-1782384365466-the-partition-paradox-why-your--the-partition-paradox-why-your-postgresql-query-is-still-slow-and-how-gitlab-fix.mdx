## Hook — The 2.5 TB elephant in the room

You have done everything right. You partitioned your table by date, added indexes, and followed the documentation. Yet that query filtering on a specific date range still crawls. The query planner stares back at you with a sequential scan across every partition. Sound familiar? The painful truth is that partitioning does not make queries faster by default — it creates *opportunities* for the planner to be faster, and those opportunities are surprisingly easy to miss. GitLab's engineering team discovered this the hard way when their ci_builds table became too wide for schema migrations, too large for effective indexing, and too critical to take offline [1]. What they learned about partition pruning validation turned their database performance around.

## Problem — When partitioning feels like a placebo

Partitioning is one of those features that looks straightforward in documentation but reveals its complexity under real-world pressure. Many developers add partitioning, see the table structure reorganize, and assume performance will follow automatically. Here is the uncomfortable truth: partitioning does not make queries faster by default — only *partition pruning* does, and pruning is not guaranteed. The most common culprit is a query that *should* benefit from pruning but does not, because the planner cannot determine the partition constraint at planning time. For example, wrapping the partition key in a function like `DATE(event_date)` instead of comparing against it directly prevents the planner from eliminating partitions [2]. Consequently, the query scans every partition, and that 100M-row table behaves exactly like it did before — just with more metadata overhead. The stakes go up from there: without pruning, indexes on individual partitions are only marginally useful, sequential scans dominate, and query times degrade linearly with data growth.

## Real-World Case — GitLab's CI database crisis

GitLab's CI/CD infrastructure scaled rapidly, and the ci_builds table grew to 2.5 TB with over 2 billion rows across 50+ columns [1]. The table was so wide and so massive that even routine operations became treacherous. Schema migrations took hours or failed outright. Index maintenance became a nightmare. Statement timeouts in production became a daily occurrence. The team validated partition pruning end-to-end, and the results were dramatic. Previously unpartitioned queries that scanned every partition now targeted a single partition. LockManager contention on replicas was eliminated entirely. The fix resolved ongoing Sev-2 incidents and opened the door for partitioning additional critical tables — ci_stages, ci_job_artifacts, and ci_pipeline_variables — to continue safely. This was not a simple "add partitioning" fix. It was a meticulous process of verifying that each piece of the optimization puzzle was actually working — starting with the EXPLAIN plan.

## Deep Dive — Reading the EXPLAIN plan like a detective

When you run `EXPLAIN (ANALYZE, BUFFERS)` on a partitioned table query, you are looking for three critical signals. First, partition pruning effectiveness: the output should show "Subplans Removed" with a non-zero value. If you see zero subplans removed, the query planner visited every partition — rendering partitioning useless for that query [6]. Second, index utilization: look for "Index Scan" versus "Seq Scan" on the individual partitions. A sequential scan on a 50 GB partition is a red flag, especially when combined with a large "rows removed by filter" count. Third, expensive operations: "Sort" with high memory usage or "Hash Aggregate" with large batches indicate the query is doing heavy lifting that a well-designed composite index could eliminate. The PostgreSQL partitioning documentation [2] emphasizes that queries benefit from partition pruning only when the WHERE clause includes the partition key in a way the planner can evaluate at *planning time* — not at execution time. This subtle distinction explains why wrapping the partition key in a function call silently disables pruning. Another common mistake: using `BETWEEN` with timestamps in a way that the planner cannot prove which partitions are relevant. Moreover, the planner's ability to prune depends on the partition constraint being explicit. Range partitioning with `LESS THAN` bounds is generally the most planner-friendly strategy.

## Workflow — A systematic diagnostic approach

When faced with a slow query on a partitioned table, follow this diagnostic workflow. Start with `EXPLAIN (ANALYZE, BUFFERS)` — never optimize blind. Check for "Subplans Removed" in the output to confirm pruning is active. If pruning is not happening, ensure the WHERE clause directly references the partition key with immutable expressions (no functions wrapping the column). Once pruning is confirmed, inspect the scan type on the accessed partitions. An "Index Scan" is good; a "Seq Scan" means you need an index that covers the WHERE clause columns. After adding a composite index, consider CLUSTER to physically reorder rows on disk — this collapses the index's visibility map and dramatically reduces random I/O for range scans. Finally, re-run EXPLAIN ANALYZE and compare the numbers. The diagram below visualizes this entire workflow as a decision tree.

## Lessons Learned — What to do differently tomorrow

This journey reveals a pattern that applies far beyond PostgreSQL: the most expensive bugs are the ones that *seem* to be fixed but are not. Testing partitioning with small datasets creates false confidence — a 1 GB table does not expose the behavior of a 2.5 TB one. The PostgreSQL indexing strategy [4] and Use The Index, Luke [7] both emphasize that partial testing is the root cause of most production database incidents. The actionable takeaway for your team: make `EXPLAIN (ANALYZE, BUFFERS)` a standard part of your query review process, not a post-incident investigation tool. Add it to your pull request template for any query that touches a partitioned table. Document the expected "Subplans Removed" count for each critical query. Run query plans against a staging environment that mirrors production data volume. Database optimization at scale is not about knowing the right answer — it is about having a reliable, reproducible process for finding it, every time. Start tomorrow by running EXPLAIN on your most expensive query. You might be surprised at what you find.

## Conclusion

The next time you stare at an EXPLAIN plan showing a sequential scan across every partition, remember GitLab's story. Partitioning is not a silver bullet — it is a tool that requires verification at every step. Add EXPLAIN (ANALYZE, BUFFERS) to your query review process today. Document the expected partition pruning behavior for your critical queries. And never assume a partition is working until you have seen the "Subplans Removed" count with your own eyes. Your future self — and the engineer on call at 3 AM — will thank you.
