> "In distributed systems, race conditions don't wait for you to finish reading this post. They're already queuing up to corrupt your checkout data."

Every e-commerce engineer has lived this nightmare: a customer adds a product at $99, hits checkout, the price updates to $79 during payment processing, and now you're either honoring the old price (losing $20) or invalidating the order (angering the customer). The solution isn't simpler code—it's smarter concurrency control.

## The Concurrency Problem

Picture this: your flash sale goes live. Three thousand customers hit checkout simultaneously. Your product table shows:

```sql
SELECT id, name, price, version, stock FROM products WHERE id = 42;
```

| id | name | price | version | stock |
|----|------|-------|---------|-------|
| 42 | Pro Widget | 99.00 | 3 | 150 |

At the exact same moment, your operations team updates the price to $79 for the sale. Without proper concurrency controls, you've got transactional chaos—some orders will use $99, others $79, and your finance team will have Questions.

## Isolation Level Strategy: SERIALIZABLE + FOR UPDATE

We recommend **SERIALIZABLE isolation** combined with `SELECT FOR UPDATE` to create an MVCC snapshot that serializes competing transactions:

```sql
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Acquire row-level lock to serialize concurrent price readers
SELECT price, version, stock 
FROM products 
WHERE id = :product_id 
FOR UPDATE;

-- Validates version matches cached version  
-- If version changed between cache read and checkout, retry transaction
WITH locked AS (
  SELECT price, version, stock 
  FROM products 
  WHERE id = :product_id 
  FOR UPDATE
)
SELECT * FROM locked 
WHERE version = :cached_version;
```

The key insight: SERIALIZABLE creates a virtual wall between transactions. When the price update commits, in-flight checkouts will see either the old snapshot (retry with fresh data) or wait for the lock and see the new price. No in-between states.

## Price Versioning: The Optimistic Lock Pattern

Instead of relying entirely on SERIALIZABLE, we add an explicit version column as our optimistic lock:

```sql
ALTER TABLE products ADD COLUMN version INTEGER NOT NULL DEFAULT 1;
CREATE INDEX idx_products_version ON products(id, version);
```

Our checkout flow validates version continuity:

```sql
-- Step 1: Read and cache on product detail page
SELECT id, name, price, version FROM products WHERE id = :product_id;

-- Store in Redis: price:{product_id} -> JSON {price: 99.00, version: 3}

-- Step 2: At checkout, validate version didn't change
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Atomic version check within transaction
UPDATE products 
SET price = price, version = version + 1 
WHERE id = :product_id 
  AND version = :cached_version
  AND stock >= :quantity
RETURNING id, price, version;
```

If `version != cached_version`, the UPDATE affects 0 rows. We catch this in application code:

```typescript
async function processCheckout(productId: string, quantity: number, cachedPrice: number, cachedVersion: number) {
  const result = await db.query(`
    UPDATE products 
    SET version = version + 1 
    WHERE id = $1 
      AND version = $2
      AND stock >= $3
    RETURNING price, version
  `, [productId, cachedVersion, quantity]);

  if (result.rowCount === 0) {
    // Version mismatch or insufficient stock—retry with fresh data
    throw new RetryableError('Price changed during checkout');
  }

  const total = result.rows[0].price * quantity;
  
  await db.query(`
    INSERT INTO orders (user_id, product_id, quantity, unit_price, total)
    VALUES ($1, $2, $3, $4, $5)
  `, [userId, productId, quantity, result.rows[0].price, total]);

  await db.query('COMMIT');
}
```

## Redis Cache Invalidation Scheme

Your price cache is only as good as your invalidation strategy. We use a multi-channel approach:

```typescript
// Price update handler (runs in admin panel service)
async function updateProductPrice(productId: string, newPrice: number) {
  await db.query(`
    UPDATE products 
    SET price = $1, 
        version = version + 1,
        updated_at = NOW() 
    WHERE id = $2
  `, [newPrice, productId]);

  // Invalidate cache immediately
  await redis.del(`price:${productId}`);

  // Publish to all application instances
  await redis.publish('price-updates', JSON.stringify({
    productId,
    newPrice,
    timestamp: Date.now()
  }));
}

// Subscribe in each application instance
async function subscribeToPriceUpdates() {
  const subscriber = redis.duplicate();
  await subscriber.connect();
  
  subscriber.subscribe('price-updates', (message) => {
    const { productId, newPrice, timestamp } = JSON.parse(message);
    console.log(`Price update: ${productId} -> $${newPrice}`);
  });
}
```

Cache key design:

- **Primary cache key**: `price:{product_id}` (current price with metadata)
- **Version cache key**: `price:{product_id}:version` (for version validation)
- **TTL**: 24 hours with sliding refresh

When price updates, the DEL ensures no stale reads. The pub/sub triggers immediate refresh in all connected instances—essential during flash sales.

## Search Index Reindexing

Your search index (Elasticsearch/Meilisearch) must reflect current prices for relevance:

```typescript
async function reindexProduct(productId: string) {
  const product = await db.query(`
    SELECT id, name, price, version

    WHERE id = $1
  `, [productId]);

  await searchIndex.upsert({
    id: productId,

    // Reindex if price moved more than 10%
    reindexReason: 'price_update'
  });
}
```

Batch reindexing during price changes prevents stale search results from showing outdated prices during checkout.

## Real-World Implementation Pattern

Here's a complete checkout service implementation:

```typescript
class CheckoutService {
  private readonly MAX_RETRIES = 3;
  private readonly RETRY_DELAY_MS = 100;

  async checkout(cart: Cart): Promise<Order> {
    for (let attempt = 1; attempt <= this.MAX_RETRIES; attempt++) {
      try {
        return await this.processCheckoutAttempt(cart);
      } catch (error) {
        if (error instanceof RetryableError && attempt < this.MAX_RETRIES) {
          await this.delay(this.RETRY_DELAY_MS * attempt);
          continue;
        }
        throw error;
      }
    }
    throw new CheckoutFailedError('Max retries exceeded');
  }

  private async processCheckoutAttempt(cart: Cart): Promise<Order> {
    const client = await db.connect();
    
    try {
      await client.query('BEGIN');
      await client.query('SET TRANSACTION ISOLATION LEVEL SERIALIZABLE');

      const items = [];
      for (const cartItem of cart.items) {
        const result = await client.query(`
          UPDATE products 
          SET version = version + 1, stock = stock - $2
          WHERE id = $1 
            AND version = $3
            AND stock >= $2
          RETURNING id, price, version, name
        `, [cartItem.productId, cartItem.quantity, cartItem.cachedVersion]);

        if (result.rowCount === 0) {
          throw new RetryableError(`Price/version changed for ${cartItem.productId}`);
        }

        items.push({
          ...result.rows[0],
          quantity: cartItem.quantity,
          total: result.rows[0].price * cartItem.quantity
        });
      }

      const orderTotal = items.reduce((sum, item) => sum + item.total, 0);
      
      const order = await client.query(`
        INSERT INTO orders (user_id, items, total, status)
        VALUES ($1, $2, $3, 'pending')
        RETURNING *
      `, [cart.userId, JSON.stringify(items), orderTotal]);

      await client.query('COMMIT');
      
      // Async cache invalidation
      await this.invalidatePriceCache(cart.items.map(i => i.productId));
      
      return order.rows[0];
    } catch (error) {
      await client.query('ROLLBACK');
      throw error;
    } finally {
      client.release();
    }
  }
}
```

## Actionable Conclusion

Here's your implementation checklist:

1. **Add version columns** to products table with default 1
2. **Use SERIALIZABLE isolation** with SELECT FOR UPDATE for checkout transactions
3. **Validate version continuity** using atomic UPDATE with version check
4. **Implement Redis cache** with DEL on price updates + pub/sub notifications
5. **Add retry logic** with exponential backoff for version mismatch failures
6. **Trigger search reindex** asynchronously after price commits

With this architecture, your checkout flow becomes idempotent—even with 10,000 concurrent customers during a flash sale, every order uses the correct price, every customer sees consistent data, and your finance team sleeps soundly.

The next time a product price changes during checkout, MVCC catches it, versioning validates it, and your customers never know the difference.
