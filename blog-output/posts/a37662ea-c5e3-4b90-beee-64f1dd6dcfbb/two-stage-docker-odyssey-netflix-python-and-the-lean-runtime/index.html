Two-Stage Docker Odyssey: Netflix, Python, and the Lean Runtime

Devops beginner dockerfile compose multi-stage 3 min read May 1, 2026

It was 3am when Netflix faced a brutal wake-up call: shipping a Python app with a demanding C extension without bloating the container. The answer was a two-stage Dockerfile that builds the C core in a builder stage and then copies only the artifacts into a lean runtime image — a pattern Netflix documented in the Netflix VMAF Dockerfile 1. This approach becomes a blueprint for modern containers: build heavy stuff once, ship a tiny battle-tested runtime 2. You’ll see how this idea translates to a Python 3.11 Flask app, with a docker-compose setup that runs alongside Redis, exposes port 5000, and keeps security tight by running as a non-root user 3.

Foundation from Netflix

Building on Netflix’s approach, multi-stage builds separate build-time artifacts from the final runtime, dramatically reducing image size and attack surface while preserving reproducibility 2. In practice, the builder stage installs compilers, compiles the C extension, and exports artifacts to a shared location for the runtime stage to consume. The runtime stage then installs just the necessary Python dependencies, copies the pre-built artifacts, switches to a non-root user for security, and configures a HEALTHCHECK to verify the app health in production 3.

Reproducibility and isolation matter here: the builder artifacts are independent of the runtime, so upgrades to the C extension or Python tooling don’t drag the entire runtime image up in size 2.

A Practical Flask + C Extension

This pattern shines when a Flask app depends on a C extension for performance. The story translates to a clean two-stage Dockerfile: the builder stage compiles the C code and collects artifacts in /out, while the runtime stage copies those artifacts, installs Python deps, and runs the app as a non-root user on port 5000 with a HEALTHCHECK. The docker-compose setup ensures Redis is available for caching or session storage, mirroring real-world microservice deployments where a lightweight API container coexists with data stores 4.

Code highlights:

  • Build-time isolation: build tools and compilers live only in the builder stage 2
  • Runtime lean: final image excludes build tools, reducing surface area 2
  • Non-root security: app runs as a dedicated non-root user 5
  • Observability: HEALTHCHECK ensures quick detection of issues in production 3

Questions you’ll validate locally include: Can the C extension compile in the builder stage? Does the Flask app respond on port 5000? Is Redis reachable from the web container? Can the health endpoint report status reliably through the HEALTHCHECK? These are the kinds of checks Docker Compose and container health checks are designed to govern 43.

Verification, Testing, and Caveats

Verification and testing become ritual: build, run, observe, adjust. Start by building and bringing up the stack with docker-compose. Then verify containers are reachable, and the health endpoint reports healthy 4. If the C extension isn’t found at runtime, confirm the /out path was populated in the builder stage and that COPY --from=builder /out /app in the runtime stage actually brings those artifacts into the final image 2.

Key commands:

  • docker-compose build
  • docker-compose up -d
  • docker-compose ps
  • curl -sS undefined
  • docker exec -u appuser web-app pytest || true # if tests exist in container

For production-grade pipelines, also consider caching strategies to speed up rebuilds and separating testing into a dedicated stage to avoid leaking test artifacts into the final image 25.

Real-World Case Study

Netflix

Netflix used a two-stage Dockerfile for the VMAF project to build the C core (libvmaf) in a builder stage and then install/copy into a lean runtime image; this pattern enables Python bindings and a production-ready container for video quality tooling.

Key Takeaway: Multi-stage builds effectively separate build-time artifacts from runtime, reducing image size and attack surface while preserving reproducibility for complex C extensions and Python integrations.

System Flow

flowchart TD A[Source Code] --> B[Builder Stage: compile C extension] B --> C[Artifacts output to /out] C --> D[Runtime Stage: lean image] D --> E[Final image with app and deps] E --> F[Docker Compose: web + redis] F --> G[HEALTHCHECK verifies /health]

Did you know? Battle scars: hidden build artifacts can sneak into final images if COPY paths mis-specify; always validate paths between builder and runtime stages.

Key Takeaways

  • Use multi-stage Dockerfiles to isolate build-time artifacts from runtime
  • Run containers as non-root users for better security
  • In production, rely on HEALTHCHECK and docker-compose for reliability
