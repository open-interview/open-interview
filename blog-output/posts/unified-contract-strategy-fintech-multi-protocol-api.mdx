> **Picture this:** It's 2 AM on a Saturday night and your fintech platform just processed 50,000 risk assessments across three different API protocols. A webhook integration team reports the REST API returned a field that doesn't exist in your GraphQL schema. Your gRPC consumers are seeing stale data. The incident root cause? Three separate contract definitions that drifted out of sync.

This is the reality of managing multi-protocol APIs without a unified contract strategy.

## The Problem: Contract Drift in Multi-Protocol Systems

In production fintech systems, exposing your risk-scoring engine across REST, GraphQL, and gRPC isn't optional—it's survival. Different client teams have different requirements: mobile teams want GraphQL's flexible queries, internal services need gRPC's performance, and webhooks require REST's ubiquity.

The traditional approach? Maintain three separate contract files. The灾难? They're never in sync.

Every field addition becomes a three-way synchronization exercise. Each deprecation requires updating OpenAPI specs, GraphQL SDL, and proto definitions independently. Migration windows get messy because you can't guarantee all protocols move together.

**The solution: Contract-driven development with a single source of truth.**

## The Contract Source: YAML-Based Unified Definition

We define our risk-scoring contract in a declarative YAML format that serves as the authoritative source. This file generates all three protocol definitions simultaneously.

```yaml
# contracts/risk-scoring/v1.yaml
name: risk-scoring
version: 1.2.0
description: Financial risk assessment and scoring API

operations:
  assess-risk:
    description: Assess financial risk for a given applicant
    versioning:
      introduced: "1.0.0"
      deprecated: "1.3.0"
      sunset: "2.0.0"
    
    rest:
      method: POST
      path: /risk/assess
      
    graphql:
      type: mutation
      field: assessRisk
      input: AssessRiskInput
      
    grpc:
      service: RiskService
      method: AssessRisk
      
    idempotency:
      required: true
      header: X-Idempotency-Key  # REST
      field: idempotencyKey      # GraphQL
      metadata: idempotency_key  # gRPC
      
    authorization:
      policy: risk.authz
      scopes: ["risk:read"]
      
    request:
      type: object
      fields:
        applicantId:
          type: string
          required: true
          validation: uuid
        amount:
          type: number
          required: true
          min: 0
        loanType:
          type: string
          required: true
          enum: [personal, business, mortgage]
        incomeVerification:
          type: boolean
          default: false
          
    response:
      type: object
      fields:
        assessmentId:
          type: string
        riskScore:
          type: integer
          min: 0
          max: 100
        riskCategory:
          type: string
          enum: [low, medium, high, critical]
        factors:
          type: array
          items:
            type: object
            fields:
              name: string
              impact: number
              description: string
        processedAt:
          type: string
          format: date-time
```

This single YAML file now drives our entire contract ecosystem.

## Cross-Protocol Versioning Strategy

Semantic versioning forms the backbone of our cross-protocol strategy. Each release bumps the shared version identifier, ensuring all protocols move in lockstep.

### Version Headers and Deprecation

We implement protocol-specific deprecation mechanisms:

**REST (HTTP Headers):**
```http
HTTP/1.1 200 OK
Sunset: Sat, 01 Jan 2027 00:00:00 GMT
Deprecation: true
Link: </v2/risk/assess>; rel="successor-version"
```

**GraphQL (@deprecated directive):**
```graphql
type Mutation {
  assessRisk(input: AssessRiskInput!): RiskAssessment
    @deprecated(reason: "Use assessRiskV2. Deprecated in v1.3.0, sunset in v2.0.0")
}
```

**gRPC (proto annotations):**
```protobuf
message RiskAssessmentRequest {
  string applicant_id = 1;
  double amount = 2;
  
  option (deprecated) = true;
  option (sunset_version) = "2.0.0";
}
```

This approach ensures clients receive clear signals across all protocols when deprecation approaches.

## Authorization with Open Policy Agent

We enforce per-operation authorization using OPA with operation-scoped rules. The policy evaluates across all protocols uniformly.

```rego
package risk.authz

default allow = false

allow {
  input.operation == "risk:assess"
  some s
  input.subject.scopes[s] == "risk:read"
}

allow {
  input.operation == "risk:assess"
  input.subject.scopes[_] == "risk:admin"
}

deny with message := "Insufficient scopes for risk:assess operation" {
  input.operation == "risk:assess"
  not allow
}
```

The policy plugs into our API gateway, receiving contextual input:

```typescript
interface AuthzInput {
  operation: string;
  subject: {
    id: string;
    scopes: string[];
  };
  protocol: 'rest' | 'graphql' | 'grpc';
  idempotencyKey?: string;
}

// Gateway middleware injects protocol and normalizes fields
const authzInput: AuthzInput = {
  operation: 'risk:assess',
  subject: { id: userId, scopes: userScopes },
  protocol: 'rest', // or 'graphql', 'grpc'
  idempotencyKey: request.headers['x-idempotency-key']
};
```

## Real-World Implementation: Contract Generation Pipeline

Our CI/CD pipeline generates protocol-specific definitions from the YAML source:

```typescript
// scripts/generate-contracts.ts
import { generateOpenAPI } from './generators/openapi';
import { generateGraphQLSDL } from './generators/graphql';
import { generateProto } from './generators/proto';
import * as yaml from 'js-yaml';
import * as fs from 'fs';

async function main() {
  const source = yaml.load(fs.readFileSync('contracts/risk-scoring/v1.yaml', 'utf8'));
  
  // Generate all three formats
  await generateOpenAPI(source, { output: 'generated/openapi/v1.yaml' });
  await generateGraphQLSDL(source, { output: 'generated/graphql/v1.schema.graphql' });
  await generateProto(source, { output: 'generated/proto/v1.proto' });
  
  // Run contract tests
  await runContractTests(source);
}
```

Generated test coverage ensures consistency:

### OpenTelemetry Observability

We instrument all three protocols with OpenTelemetry for distributed tracing:

```typescript
import { trace, SpanStatusCode } from '@opentelemetry/api';

const tracer = trace.getTracer('risk-scoring-service');

async function assessRisk(input: AssessRiskInput): Promise<RiskAssessment> {
  return tracer.startActiveSpan('risk:assess', async (span) => {
    try {
      span.setAttribute('risk.operation', 'assess');
      span.setAttribute('risk.protocol', 'graphql');
      
      // Add idempotency tracking
      if (input.idempotencyKey) {
        span.setAttribute('risk.idempotency_key', input.idempotencyKey);
      }
      
      const result = await riskEngine.calculate(input);
      span.setAttribute('risk.score', result.riskScore);
      span.setStatus({ code: SpanStatusCode.OK });
      return result;
    } catch (error) {
      span.setStatus({ code: SpanStatusCode.ERROR, message: error.message });
      throw error;
    } finally {
      span.end();
    }
  });
}
```

Distributed traces now flow across REST, GraphQL, and gRPC endpoints with consistent attribute keys.

## Migration Strategy: Contract-Driven Development

When migrating between contract versions, we generate adapters automatically:

```typescript
// adapters/v1-to-v2.ts
import { map } from 'functools';

export class RiskAssessmentV1ToV2Adapter {
  adaptV1ToV2(request: RiskAssessmentV1Request): RiskAssessmentV2Request {
    return {
      ...map(request, {
        applicant_id: 'applicantId',
        income_verification: 'incomeVerification'
      }),
      assessmentVersion: 'v2',
      enhancedFactors: true
    };
  }
}
```

Clients running older protocol versions automatically receive adapted requests, ensuring seamless migrations.

## Conclusion

The unified contract strategy transforms multi-protocol API management from a synchronization nightmare into a declarative, reproducible workflow. The key takeaways:

1. **Single source of truth**: Define contracts in YAML, generate OpenAPI, GraphQL SDL, and proto files automatically.

2. **Cross-protocol versioning**: Use semantic versioning with protocol-specific deprecation headers to signal migration windows clearly.

3. **Authorization at the policy layer**: Implement OPA policies that evaluate uniformly across REST, GraphQL, and gRPC.

4. **Contract tests from the source**: Generate consistency tests that verify field parity across all three protocols.

5. **Observability as a first-class concern**: Instrument all protocols with OpenTelemetry for end-to-end distributed tracing.

Your incident at 2 AM becomes a thing of past—when contract changes require updating a single YAML file, not three drifting definitions.

**Start small:** Choose one operation, define it in YAML, generate all three formats, and run the consistency tests. Then scale across your entire API surface.
