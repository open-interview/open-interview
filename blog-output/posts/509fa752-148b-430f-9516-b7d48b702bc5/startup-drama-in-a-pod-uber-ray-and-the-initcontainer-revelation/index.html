Fun fact: Many teams discover startup coordination problems only after production incidents push Ray or other distributed workloads into fragile boot sequences.

Picture this: Uber’s journey to Ray on Kubernetes began with startup drama: Ray workers needed the head node address in a volatile host-networked cluster [1]. Init containers stepped in to bootstrap before the main app, decoupling bootstrap logic from runtime. This pattern reshaped how teams think about reliability, scale, and bootstrap complexity.

Problem at the Edge

Picture this: distributed ML workloads waking up in a cluster where ports shift and networks bend. Init containers solve the early choreography problem by running to completion before the main app starts [2]. They must exit successfully (exit code 0) for the pod to proceed, and then the main containers take the stage. This upfront discipline matters when environment discovery, migrations, or secret fetches must happen before any work begins [3].

Discovery

This is where the story shifts from bootstrap to runtime. Init containers execute sequentially, each one finishing before the next begins, and they share pod resources without contributing to the steady-state footprint [2]. In contrast, sidecar containers run continuously alongside the main application for the entire pod lifetime, sharing network namespaces, volumes, and IPC, and they can restart independently [4]. The result is a clean separation: bootstrap logic lives in init containers, while ongoing concerns live in sidecars [5].

Code example:

apiVersion: v1
kind: Pod
metadata:
  name: webapp
spec:
  initContainers:
  - name: migrate-db
    image: migrate:latest
    command: ['python', 'migrate.py']
  - name: fetch-config
    image: config-fetcher:latest
    volumeMounts:
    - name: config
      mountPath: /etc/config
  containers:
  - name: webapp
    image: webapp:latest
    volumeMounts:
    - name: config
      mountPath: /etc/config
  - name: log-shipper
    image: fluentd:latest
    volumeMounts:
    - name: logs
      mountPath: /var/log

This code demonstrates the split: initContainers run first, then the main app starts, with a sidecar handling logging during the pod lifetime [2][4].

The Twist

Counterintuitive insight: init containers share pod resources, but don’t contribute to the ongoing resource usage, while sidecars add to the pod’s total resource requests and limits and often provide lifecycle services like logging or configuration reloads [5]. This nuance matters when sizing a pod for a production workload: bootstrap tasks can be generous early on, but steady-state capacity must cover the sidecar’s continuous presence [6]. Moreover, sidecars can fail and restart independently without toppling the main app, unless careful coupling or hard-fail policies are in place [3].

Real-World Proof

Many teams discover that the right split unlocks reliability in distributed systems. In Uber’s Ray-on-Kubernetes journey, init containers provided reliable startup coordination and environment discovery, decoupling bootstrap logic from the distributed workload and enabling scalable, reliable startup for Ray workers in environments with host networking and dynamic ports [1]. This approach aligns with Kubernetes guidance on separating bootstrap work from runtime work and using init containers for preflight checks, migrations, and secret fetches [2][3].

The Payoff

When to use init containers vs sidecars becomes a design decision about startup vs. runtime needs. Use init containers for bootstrap tasks: migrations, configuration generation, secret retrieval, and preflight checks that must complete before the app runs [3]. Use sidecars for ongoing concerns: logging, monitoring, proxies, configuration reloaders, or backup processes that must live with the app for its entire lifetime [4].

Wrapping Up

Start with init containers to coordinate bootstrap; layer in sidecars for ongoing needs. Review patterns as workloads grow and port dynamics change. Uber’s Ray journey shows bootstrap and runtime separation isn’t optional; it’s a design choice that saves stability under pressure.
