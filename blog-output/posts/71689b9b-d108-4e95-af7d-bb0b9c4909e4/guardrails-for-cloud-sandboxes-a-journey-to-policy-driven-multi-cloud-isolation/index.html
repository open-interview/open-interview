Guardrails for Cloud Sandboxes: A Journey to Policy-Driven, Multi-Cloud Isolation

Terraform intermediate terraform 4 min read May 1, 2026

Capital One's real-world push into policy-as-code governance across Terraform Cloud shows what happens when plans are checked before they touch production 1. Picture this: a SaaS platform deploying customer sandboxes across AWS and GCP from a single monorepo, hunting for a scalable, provider-agnostic network core, and craving strict per-customer isolation. The stakes are clear: misconfigurations slide into prod, tenants collide, and audits become cliffhangers. This story traces the path teams take to turn that risk into a repeatable, auditable process.

HOOK & Stakes

Problem lurks in quiet corners: a shared root module provisioning a provider-agnostic network core (VPC/VNet) must scale to many customers, across clouds, without stepping on each other. You need per-customer backends with unique keys, per-customer IAM/policies expressed as code, and a CI gate that stops any plan from touching other tenants or production. Capital One's journey demonstrates the power of encoding governance as code and gating deployments early 1. This leads to a narrative where failure is not a fate but a design constraint.

THE CORE IDEA

Building on the opening premise, the first discovery is a provider-agnostic network core module that can be instantiated per customer using for_each. This enables a single source of truth for network design while keeping each tenant isolated. The skeleton plan hinges on provider aliases for AWS and GCP, with each customer binding to the correct provider context. In practice, this means per-customer backends keyed by customer_id, so state lives in distinct places and access remains scoped. The architecture is reinforced by policy-as-code gates that enforce IAM boundaries and tenancy rules before any plan reaches apply.

Policy-as-Code in Practice

Per-customer IAM and policies get expressed as code, not as handwritten guardrails. OPA provides policy enforcement as a gate, while Sentinel offers an alternative governance layer that can enforce plan-time and apply-time checks. The goal is to encode who can do what, on which tenants, and under what conditions, then run these checks in CI before any Terraform plan or apply proceeds. Tests with simulators catch misconfigurations early, reducing the blast radius when onboarding new customers or adding new tenancy models 1.

GATEKEEPING THE CI PIPELINE

The CI gate sits between plan and apply. When a plan runs, the gate extracts the intent (which customers, which backends, which regions) and validates that no cross-tenant or prod touch is attempted. If the gate detects any leakage, it fails fast, with a prescriptive error that points to the offending tenant or resource. This requires a small, self-contained gate script and a Terraform-friendly policy pack (OPA or Sentinel) that can evaluate plan data, tenant scoping, and environment boundaries. The skeleton below shows a minimal CI gate in action, relying on provider aliases and per-customer keys to keep scope tight.

REAL-WORLD PROOF

Reality check comes from Netflix, a pioneer in chaos engineering. Simian Army experiments reveal how failure modes ripple through complex systems and why preemptive governance and testing matter. The public GitHub repository documents the approach, milestones, and lessons learned from building resilient systems at scale 2.

TRANSFORMATION & TAKEAWAYS

Building resilience means turning governance into code, and code into a repeatable process. The journey shows four practical outcomes: (1) provider-agnostic core module design that scales across clouds; (2) per-tenant backends to guarantee isolation; (3) policy-as-code that codifies access and scope; (4) a CI gate that prevents cross-tenant or prod touch. The outcome is a deployment that feels like a well-rehearsed play rather than a roll of the dice. The next chapter is implementation in your own repo, starting with a minimal core and an incremental policy pack.

Real-World Case Study

Capital One

Capital One implemented policy-as-code governance for Terraform Cloud using Sentinel to enforce security and compliance across a large, multi-team infrastructure footprint. Policies were stored in a centralized repository and enforced between Terraform plan and apply, with automated testing and version-controlled policy sets guiding deployments across multiple environments.

Key Takeaway: Encode governance as code and gate plans early in CI/CD; version-control policies; test policies with simulators before applying; centralize policy management to scale across teams and environments.

Did you know? Chaos is not random; it’s a design discipline. Organizations that test failure scenarios early ship more reliable systems.

Key Takeaways

  • Use provider aliases for AWS and GCP to enable multi-cloud operations
  • Isolate tenants with per-customer backends and unique keys
  • Encode IAM/policies as code (OPA/Sentinel) and gate plans early
  • CI gate checks plans do not touch other tenants or prod
