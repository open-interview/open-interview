The Burst That Revealed the Hidden Cache War

Linux advanced linux 5 min read April 26, 2026

It was a 50-server Microsoft production cluster—the OneRF lineage—that first whispered the truth: tail latency can spike during bursts even when CPU, memory, and disk I/O look calm. The moment the burst hit, P99 latency climbed, and the clock kept ticking while metrics stayed pristine. This real-world tension—between cache pressure and memory allocators—remade the approach to diagnosis and mitigation, showing that cache behavior can drive tail latency 1.

Context: The Burst and the Hidden Cache War

Many developers discover that tail latency is not always a CPU bottleneck or a disk I/O bottleneck. In practice, the culprit often hides in memory management and caching strategies inside the kernel and in container boundaries. Building on this Microsoft case study, tail latency can be dramatically affected by how memory and cache are allocated and partitioned across backends 1. This section sets the stage by reframing the problem: bursts illuminate the fault lines where cache pressure and slab fragmentation collide with container boundaries, triggering latency tails rather than average-case slowness. The stakes are real: a few milliseconds of tail latency can ripple into customer-visible delays and degraded service perception.

The Safe Mitigations, With Observability Intact

The core idea is to tune memory subsystem knobs and enforce sane cache and memory boundaries without sacrificing visibility into ongoing workloads. Start with kernel memory tuning, then move to container-level constraints as needed. Here are practical steps and the reasoning behind them:

  • Reduce page cache pressure to prevent evictions from starving cache-backed backends. Command: sysctl -w vm.vfs_cache_pressure=60 (lower than default 100 but still conservative). This helps keep hot file data in memory for repeated access, reducing tail latency swings during bursts 4.
  • Increase headroom for free memory by modestly raising min_free_kbytes, ensuring that sudden cache pressure does not trigger cascading evictions 4. Command: sysctl -w vm.min_free_kbytes=262144.
  • Release cached pages temporarily to assess whether cache pressure is the controlling factor without permanent disruption. Command: sync; echo 3 > /proc/sys/vm/drop_caches (requires maintenance window; use carefully) 5.
  • If containers are the source of pressure, adjust their memory ceilings to prevent a single hungry container from starving others. Docker example: docker update --memory=4g --memory-swap=6g <container_id>; Kubernetes example: kubectl set resources deployment <name> --limits memory=4Gi --requests memory=2Gi 1112.

Consequence: observability remains intact because all mitigations are designed to be measured with the same toolchain. The diagnostic trio—iostat, vmstat, slabtop—continues to be run after each change to confirm whether the tail latency improvement persists 678.

Putting It All Together

The journey starts with a real-world tension: bursts reveal how cache behavior shapes tail latency. By methodically verifying cache pressure, identifying heavy memory users, and applying conservative mitigations with observability preserved, teams can reduce P99 latency without sacrificing system visibility. The lesson travels beyond a single cluster: dynamic, workload-aware cache management can be the difference between a smooth slate of average latency and tail latency that slices through user experience. As the case study shows, the right partitioning of cache—backed by disciplined instrumentation—transforms chaos into predictability 1.

Wrapping Up

The journey reveals a timeless truth: performance is a system property. Bursts expose the fault lines, but disciplined instrumentation, conservative mitigations, and clear observability turn volatility into predictability. The next time a burst arrives, the team will know where to listen to the cache and how to reply with measured, safe changes.
