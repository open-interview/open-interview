A NUMA Tale: Unraveling Tail Latency with Linux Memory Reclaim

Linux intermediate linux 5 min read April 27, 2026

In 2013, LinkedIn faced intermittent tail latency spikes on NUMA servers during peak ingestion for an online-graph workload. Standard metrics looked healthy, yet tail latency persisted, traced to how Linux memory reclaim interacts with NUMA locality 1. This journey walks through a concrete, real-world diagnostic workflow using only default Linux tools to confirm reclaim pressure, identify the NUMA node(s) and processes responsible, and apply safe mitigations that don’t disturb other tenants.

Context: The problem that refuses to quit

Many developers confront the paradox: a system appears healthy in aggregate, but tail latency stubbornly lingers. Building on LinkedIn's experience that zone reclaim on NUMA systems can dominate tail latency 1, the goal is to verify whether kernel memory reclaim is pulling the tail, and if so, where the pressure is originating. The first step is to observe memory activity in real time and spot signs that reclaim is competing with app memory. The following command sequences establish baseline visibility and hint at reclamation pressure.

Journey: Mapping memory and NUMA locality

Building on the initial signal, the next discovery path maps memory across NUMA nodes and surfaces where reclamation pressure concentrates. Start with a top-down view of NUMA topology and per-node memory distribution. This reveals constrained nodes and potential hotspots that reclamation may be exploiting.

{
  "type": "code",
  "raw": "```\nnumactl --hardware\n```",
  "lang": "",
  "text": "numactl --hardware"
}

And inspect per-node memory statistics directly:

{
  "type": "code",
  "raw": "```\nfor n in /sys/devices/system/node/node*/meminfo; do echo \"=== $n ===\"; cat $n; done\n```",
  "lang": "",
  "text": "for n in /sys/devices/system/node/node*/meminfo; do echo \"=== $n ===\"; cat $n; done"
}

These outputs identify how memory is allocated across nodes and whether some nodes are under memory pressure. Understanding current NUMA binding policies helps predict how memory pressure may cascade across tenants. Verify the current NUMA binding policy:

{
  "type": "code",
  "raw": "```\nnumactl --show\n```",
  "lang": "",
  "text": "numactl --show"
}

Finally, list memory-heavy processes to understand who might be driving demand:

{
  "type": "code",
  "raw": "```\nps aux --sort=-rss | head -20\n```",
  "lang": "",
  "text": "ps aux --sort=-rss | head -20"
}

Together, these steps reveal whether tail latency correlates with NUMA node pressure and which processes are the loudest memory consumers, setting up targeted mitigations. Transition: with the NUMA map in hand, the next move is to tailor mitigation without broad disruption.

Twist: The art of safe tuning without collateral damage

A surprising twist often emerges: the urge to blunt memory pressure with aggressive defaults can swing the balance against other tenants, especially in multi-tenant environments. The counterintuitive insight is that safety comes from targeted, conservative changes rather than sweeping rewrites. The story shifts to three precise mitigations that address reclaim pressure while preserving fairness and predictability.

  1. Increase vm.min_free_kbytes conservatively (roughly 1% of total RAM). This helps keep pages readily reclaimable without triggering excessive paging under peak load.
{
  "type": "code",
  "raw": "```\nTOTAL_KB=$(grep MemTotal /proc/meminfo | awk '{print $2}')\nMIN_FREE_KB=$((TOTAL_KB / 100))\nsudo sysctl -w vm.min_free_kbytes=${MIN_FREE_KB}\n```",
  "lang": "",
  "text": "TOTAL_KB=$(grep MemTotal /proc/meminfo | awk '{print $2}')\nMIN_FREE_KB=$((TOTAL_KB / 100))\nsudo sysctl -w vm.min_free_kbytes=${MIN_FREE_KB}"
}
  1. Tune vm.vfs_cache_pressure to 200 to favor reclaim of inode/dentry caches when memory is tight, reducing cache-related stalls:
{
  "type": "code",
  "raw": "```\nsudo sysctl -w vm.vfs_cache_pressure=200\n```",
  "lang": "",
  "text": "sudo sysctl -w vm.vfs_cache_pressure=200"
}
  1. Pin critical ingestion/processing processes to isolated NUMA nodes using either numactl or taskset, so that important workloads don’t compete with less critical tenants for memory.
{
  "type": "code",
  "raw": "```\n# Example: bind a critical PID 1234 to NUMA node 0\nsudo numactl --cpunodebind=0 --membind=0 -p 1234\n# Or launch a critical process on a specific node\nsudo numactl --cpunodebind=0 --membind=0 /path/to/critical-service --args\n```",
  "lang": "",
  "text": "# Example: bind a critical PID 1234 to NUMA node 0\nsudo numactl --cpunodebind=0 --membind=0 -p 1234\n# Or launch a critical process on a specific node\nsudo numactl --cpunodebind=0 --membind=0 /path/to/critical-service --args"
}

These steps are deliberately cautious, designed to avoid regressing other workloads. After applying changes, verify observability to ensure performance improves without regressions.

If performance did not improve or worsened for other workloads, revert the changes or re-balance pinning to less aggressive bindings. The discipline here is incremental change and close observation. The LinkedIn case study reinforces this approach: careful VM tuning and NUMA-aware process affinity can yield latency improvements without broad disruption 1.

Real-world proof: LinkedIn’s war story

The opening hook isn’t fiction. LinkedIn’s production environment faced intermittent latency bursts during peak ingestion on NUMA servers, driven by how Linux memory reclamation interacts with NUMA locality. The lesson: zone reclaim can dominate tail latency in latency-sensitive, multi-tenant workloads, and observability with native tools can illuminate a path to meaningful improvements without sweeping changes 1. This war story anchors the practical workflow described above, highlighting the delicate balance between reclaim behavior and NUMA locality. The actionable takeaway is a disciplined, data-first approach that starts with visibility and ends with targeted, safe tuning rather than broad sweeps. Inline citation 1 anchors this narrative to the source material.

Conclusion: What to do tomorrow

tail latency in NUMA-heavy, multi-tenant environments often hides in plain sight—behind kernel memory reclamation and node topology. The three-part workflow—confirm reclaim pressure, identify NUMA nodes and processes, and apply conservative, targeted mitigations—offers a pragmatic path forward. Start by verifying reclaim indicators with vmstat and /proc/vmstat, map NUMA topology, and then apply minimal, reversible tunings to vm.min_free_kbytes and vm.vfs_cache_pressure, while isolating critical processes. The discipline of observability remains the strongest ally: continuous monitoring should be the default, not an afterthought. A memorable takeaway to share with the team: small, deliberate VM tuning guided by NUMA awareness can unlock dramatic latency improvements without destabilizing multi-tenant workloads.
