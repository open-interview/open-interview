> **Picture this:** It's 2 AM. Your staging deployment just failed health checks after a "routine" nightly build. The on-call engineer scrambles to identify which artifact was deployed, which commit caused the failure, and how to roll back—all while users report issues. Sound familiar? This is the reality of managing heterogeneous CI/CD ecosystems.

In enterprise environments, it's common to have multiple CI tools serving different purposes: GitHub Actions for rapid PR feedback, Jenkins for resource-intensive nightly builds, and GitLab CI for deployment orchestration. But without a unified strategy, you're heading toward operational chaos.

This guide walks through designing a cross-tool CI workflow that brings **consistency**, **traceability**, and **reliability** to your pipeline ecosystem.

## The Cross-Tool Architecture

At the core of this architecture lies a **centralized, immutable artifact registry** alongside a **single cross-tool manifest** that serves as the source of truth for every deployment.

```
┌─────────────────────────────────────────────────────────────────┐
│                    CROSS-TOOL PIPELINE                          │
├─────────────────────────────────────────────────────────────────┤
│  GitHub Actions      Jenkins            GitLab CI              │
│  ───────────────    ────────           ───────────             │
│  PR Validation   →  Nightly Builds  →  Staging Deploy         │
│  (fast feedback)     (heavy loads)      (controlled rollout)   │
└─────────────────────────────────────────────────────────────────┘
              ↓                    ↓                    ↓
    ┌──────────────────────────────────────────────┐
    │     Centralized Immutable Artifact Registry │
    │     (e.g., Artifactory, GHCR, GitLab Registry)│
    └──────────────────────────────────────────────┘
              ↓
    ┌──────────────────────────────────────────────┐
    │     Cross-Tool Manifest (JSON/YAML in Redis) │
    └──────────────────────────────────────────────┘
```

Each pipeline stage references this manifest for artifact locations and metadata, ensuring every system operates from identical information.

## Correlation ID: The Thread Connecting Everything

A **correlation ID** (also known as trace ID or request ID) propagates through every pipeline stage, enabling end-to-end traceability. Here's how to implement it:

### GitHub Actions: PR Validation Stage

```yaml
# .github/workflows/pr-validation.yml
name: PR Validation

on:
  pull_request:
    branches: [main]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - name: Generate Correlation ID
        id: correlation
        run: |
          CORRELATION_ID="pr-$(date +%s)-${{ github.event.pull_request.number }}"
          echo "correlation_id=$CORRELATION_ID" >> $GITHUB_OUTPUT
          echo "CORRELATION_ID=$CORRELATION_ID" >> $GITHUB_ENV

      - name: Build and Test
        env:
          CORRELATION_ID: ${{ steps.correlation.outputs.correlation_id }}
        run: |
          ./gradlew build --no-daemon
          
      - name: Publish to Immutable Registry
        env:
          CORRELATION_ID: ${{ steps.correlation.outputs.correlation_id }}
        run: |
          ARTIFACT_SHA=$(sha256sum app.jar | cut -d' ' -f1)
          docker tag app:latest registry.example.com/app:${CORRELATION_ID}
          docker push registry.example.com/app:${CORRELATION_ID}
          
          # Record in manifest
          curl -X POST $MANIFEST_SERVICE/entries \
            -H "Content-Type: application/json" \
            -d "{
              \"correlation_id\": \"$CORRELATION_ID\",
              \"artifact_sha\": \"$ARTIFACT_SHA\",
              \"commit_sha\": \"${{ github.sha }}\",
              \"source\": \"github-actions\",
              \"stage\": \"pr-validation\"
            }"
```

### Jenkins: Nightly Heavy Builds

```groovy
// Jenkinsfile
pipeline {
    agent any
    
    environment {
        CORRELATION_ID = "${params.CORRELATION_ID ?: "nightly-${env.BUILD_NUMBER}"}"
    }
    
    stages {
        stage('Validate Manifest Entry') {
            steps {
                script {
                    def entry = httpRequest(
                        url: "${MANIFEST_SERVICE}/entries/${env.CORRELATION_ID}",
                        validResponseCodes: '200,404'
                    )
                    if (entry.status == 200) {
                        echo "Correlation ID ${env.CORRELATION_ID} already processed - skipping"
                        currentBuild.result = 'SUCCESS'
                        return
                    }
                }
            }
        }
        
        stage('Heavy Build') {
            steps {
                sh '''
                    ./gradlew fullBuild --no-daemon
                    ARTIFACT_SHA=$(sha256sum build/libs/app.jar | cut -d' ' -f1)
                    docker tag app:latest registry.example.com/app:${CORRELATION_ID}
                    docker push registry.example.com/app:${CORRELATION_ID}
                '''
            }
        }
        
        stage('Update Manifest') {
            steps {
                sh """
                    curl -X PUT \$MANIFEST_SERVICE/entries/\$CORRELATION_ID \\
                        -H "Content-Type: application/json" \\
                        -d '{
                            "correlation_id": "${env.CORRELATION_ID}",
                            "artifact_sha": "'$ARTIFACT_SHA'",
                            "commit_sha": "'${env.GIT_COMMIT}'",
                            "source": "jenkins",
                            "stage": "nightly-build",
                            "dependencies": ${getDependencyVersions()}
                        }'
                """
            }
        }
    }
}
```

### GitLab CI: Staging Deployment

```yaml
# .gitlab-ci.yml
stages:
  - deploy
  - health-check
  - rollback

variables:
  MANIFEST_SERVICE: "https://manifest-service.example.com"
  REGISTRY: "registry.example.com"

deploy:staging:
  stage: deploy
  image: alpine:latest
  before_script:
    - apk add --no-cache curl docker
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $REGISTRY
  script:
    - |
      CORRELATION_ID="${CI_COMMIT_REF_NAME}-${CI_PIPELINE_ID}"
      echo "CORRELATION_ID=$CORRELATION_ID" >> deploy.env
      
      # Fetch artifact from registry
      docker pull $REGISTRY/app:$CORRELATION_ID
      docker tag $REGISTRY/app:$CORRELATION_ID app:latest
      
      # Deploy to staging
      kubectl set image deployment/app app=app:latest --record
      
      # Record deployment in manifest
      curl -X PUT $MANIFEST_SERVICE/entries/$CORRELATION_ID \
        -H "Content-Type: application/json" \
        -d "{
          \"correlation_id\": \"$CORRELATION_ID\",
          \"stage\": \"staging-deploy\",
          \"deployed_by\": \"gitlab-ci\"
        }"
  artifacts:
    reports:
      dotenv: deploy.env

healthcheck:staging:
  stage: health-check
  image: curlimages/curl:latest
  script:
    - |
      source deploy.env
      HEALTH_RESPONSE=$(curl -s -w "\n%{http_code}" \
        "https://staging.example.com/health?x-correlation-id=$CORRELATION_ID")
      HTTP_CODE=$(echo "$HEALTH_RESPONSE" | tail -n1)
      
      if [ "$HTTP_CODE" != "200" ]; then
        echo "Health check failed with HTTP $HTTP_CODE"
        # Trigger unified rollback
        curl -X POST "$MANIFEST_SERVICE/rollback" \
          -H "Content-Type: application/json" \
          -d "{
            \"correlation_id\": \"$CORRELATION_ID\",
            \"environment\": \"staging\",
            \"reason\": \"health_check_failure\"
          }"
        exit 1
      fi
```

## Enforcing Idempotence

Idempotence ensures that re-running a pipeline stage with the same inputs produces identical results. This is critical for retries and recovery scenarios.

### Content-Addressed Storage

Using content-addressed storage for artifacts means the artifact URL itself contains the SHA256 hash:

```
registry.example.com/app/sha256:abc123...def456
```

If you attempt to push an identical artifact, the registry detects the duplicate and returns success without re-uploading.

### State-Based Pipeline Execution

Persist pipeline state externally, keyed by correlation ID:

```python
# manifest-service/pipeline_state.py
from redis import Redis
import json

redis_client = Redis.from_url(os.environ['REDIS_URL'])

def is_stage_complete(correlation_id: str, stage: str) -> bool:
    key = f"pipeline:state:{correlation_id}:{stage}"
    return redis_client.exists(key)

def mark_stage_complete(correlation_id: str, stage: str, metadata: dict):
    key = f"pipeline:state:{correlation_id}:{stage}"
    redis_client.set(key, json.dumps(metadata), ex=86400)  # 24h TTL
```

Each pipeline stage checks this state before execution:

```yaml
# Idempotent stage example in GitHub Actions
- name: Check if stage already completed
  id: check-state
  run: |
    STATE=$(curl -s $MANIFEST_SERVICE/state/$CORRELATION_ID/pr-validation)
    if [ "$STATE" = "complete" ]; then
      echo "skip=true" >> $GITHUB_OUTPUT
    fi

- name: Build (skip if already complete)
  if: steps.check-state.outputs.skip != 'true'
  run: ./gradlew build
```

## Unified Rollback: The Safety Net

When health checks fail, the unified rollback path ensures consistent recovery:

```python
# manifest-service/rollback.py
@app.route('/rollback', methods=['POST'])
def rollback():
    data = request.json
    correlation_id = data['correlation_id']
    environment = data['environment']
    
    # Fetch previous known-good entry
    current_entry = fetch_manifest_entry(correlation_id)
    previous_entry = fetch_previous_good_entry(
        environment=environment,
        before=current_entry['timestamp']
    )
    
    # Promote previous artifact
    promote_artifact(
        source_sha=previous_entry['artifact_sha'],
        target_tag=f"rollback-{int(time.time())}",
        registry=REGISTRY
    )
    
    # Trigger health validation
    trigger_health_check(
        environment=environment,
        correlation_id=f"rollback-{correlation_id}",
        artifact_sha=previous_entry['artifact_sha']
    )
    
    return {"status": "rollback_initiated", "previous_sha": previous_entry['artifact_sha']}
```

This approach guarantees that:
1. **The manifest is always the source of truth** — no guessing which version is "good"
2. **Artifacts are immutable** — we never overwrite, we always promote
3. **Rollback is consistent** — the same process works regardless of which CI tool triggered it

## Conclusion: Bringing Order to CI Diversity

Implementing a cross-tool CI workflow isn't about replacing your existing tools—it's about adding a **coordination layer** that brings consistency:

- **Correlation IDs** provide end-to-end traceability across GitHub Actions, Jenkins, and GitLab CI
- **Immutable artifact registries** ensure reproducibility and enable safe rollbacks
- **Idempotent pipelines** make retries safe and efficient
- **Centralized manifest** serves as the single source of truth for every deployment

Start small: pick one correlation ID format, establish your manifest storage, and instrument one pipeline. Then expand iteratively.

The goal isn't perfect uniformity—it's operational confidence in a diverse tooling landscape.
