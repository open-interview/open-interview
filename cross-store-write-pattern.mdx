> "In distributed systems, the hard part isn't making things work—it's making them work reliably when things fail."

Every engineer who's built a multi-tenant SaaS platform has faced this moment: a user makes an API call, quota gets decremented in DynamoDB, the counter increments in Redis, but Cassandra fails to write the audit log. Now you have inconsistent state, no record of what happened, and a support ticket waiting in your inbox.

This is the reality of cross-store writes. You can't use traditional ACID transactions because you're spanning multiple databases with different consistency models. In this post, I'll walk you through a battle-tested pattern for exactly this scenario—one that's used in production at scale.

## The Problem: Quota Consumption Across Four Stores

Imagine your API gateway handles tenant requests. Each request consumes quota. Here's your data topology:

| Store | Purpose | Consistency Model |
|-------|---------|-------------------|
| **MongoDB** | Tenant configurations | Strong |
| **Redis** | Real-time counters | Eventual |
| **DynamoDB** | Daily quotas | Strong (conditional) |
| **Cassandra** | Audit logs | Eventual |

The challenge: when an API call consumes quota, you need to atomically:
1. **Decrement** quota in DynamoDB (with conditional check: `quota_remaining > 0`)
2. **Increment** `quota_used` in Redis
3. **Append** audit row in Cassandra

Failure at any step creates inconsistency. And here's the kicker—network calls fail. Always. Your design must account for it.

## Solution: Deterministic Transaction IDs with Compensating Writes

The core insight is simple: **make every operation idempotent using a deterministic transaction ID**, then handle failures through compensating transactions.

### Step 1: Generate a Unique Transaction ID

```javascript
const txId = `${tenantId}:${requestId}:${timestamp}`;
```

This composite key guarantees uniqueness and enables deduplication. Store it in Redis with a TTL matching your quota window (typically 24 hours).

### Step 2: Check Idempotency Before Starting

```javascript
async function checkIdempotency(redis, txId) {
  const existing = await redis.get(`tx:${txId}:status`);
  if (existing) {
    return JSON.parse(existing); // Return cached result, skip processing
  }
  return null;
}
```

If the transaction already completed, return the cached result immediately. This is critical for retries and replay scenarios.

### Step 3: DynamoDB Conditional Decrement

```javascript
const dynamoResult = await dynamoClient.update({
  TableName: 'tenant_quotas',
  Key: { tenantId },
  UpdateExpression: 'SET quota_remaining = quota_remaining - :consume',
  ConditionExpression: 'quota_remaining > :zero',
  ExpressionAttributeValues: {
    ':consume': 1,
    ':zero': 0
  },
  ReturnValues: 'ALL_NEW'
}).promise();
```

The `ConditionExpression` is your safety net—it ensures quota doesn't go negative. If quota is exhausted, DynamoDB throws a `ConditionalCheckFailedException`, and you skip the remaining steps.

### Step 4: Redis Increment

```javascript
await redis.multi()
  .incr(`quota:used:${tenantId}`)
  .set(`tx:${txId}:step:dynamo`, 'complete', 'EX', 86400)
  .exec();
```

Using a Redis transaction (`MULTI`/`EXEC`) ensures atomicity. Mark the DynamoDB step complete so replays know where to resume.

### Step 5: Cassandra Audit Append

```javascript
await cassandra.execute(
  `INSERT INTO audit_logs (tx_id, tenant_id, action, quota_before, quota_after, timestamp)
   VALUES (?, ?, ?, ?, ?, ?)`,
  [txId, tenantId, 'quota_consumed', 
   dynamoResult.Attributes.quota_remaining + 1,
   dynamoResult.Attributes.quota_remaining,
   Date.now()]
);
```

Using `tx_id` as the partition key enables deduplication on replay. Cassandra's eventual consistency works in your favor here—writes are fast, and retries eventually succeed.

### Step 6: Mark Transaction Complete

```javascript
await redis.set(
  `tx:${txId}:status`,
  JSON.stringify({ status: 'success', quota_remaining: dynamoResult.Attributes.quota_remaining }),
  'EX', 86400
);
```

Store the final result with a TTL matching your quota reset period.

## Failure Handling: The Compensating Transaction Pattern

What happens when Cassandra fails after DynamoDB and Redis already updated? You need a compensating write:

```javascript
async function handleCassandraFailure(txId, tenantId, dynamoResult) {
  // Compensating write: restore DynamoDB quota
  await dynamoClient.update({
    TableName: 'tenant_quotas',
    Key: { tenantId },
    UpdateExpression: 'SET quota_remaining = quota_remaining + :restore',
    ExpressionAttributeValues: { ':restore': 1 }
  }).promise();

  // Decrement Redis (it already incremented)
  await redis.decr(`quota:used:${tenantId}`);

  // Queue for retry via dead-letter mechanism
  await sendToDLQ({
    txId,
    tenantId,
    failedStep: 'cassandra',
    originalRequest: /* ... */
  });
}
```

### Dead-Letter Queue Strategy

For permanent failures, push to a DLQ:

```javascript
async function sendToDLQ(message) {
  await sqs.sendMessage({
    QueueUrl: process.env.DLQ_URL,
    MessageBody: JSON.stringify({
      ...message,
      retryCount: message.retryCount || 0,
      failedAt: new Date().toISOString()
    })
  });
}
```

A background worker processes the DLQ with exponential backoff, retrying Cassandra writes up to 5 times before alerting on-call engineers.

## Optimized Write Order

The sequence matters. Here's the optimal order with rationale:

1. **Idempotency check** (Redis) — Fastest failure, cheapest to reject duplicates
2. **DynamoDB conditional decrement** — Most critical, prevents over-consumption
3. **Redis increment** — Fast, in-memory, easy to compensate
4. **Cassandra audit** — Can be retried via DLQ, lowest priority

This order minimizes blast radius: if Redis fails, DynamoDB hasn't changed. If Cassandra fails, you can compensate via DynamoDB restoration.

## Schema Reference

**DynamoDB - tenant_quotas table:**
```json
{
  "tenantId": "String (PK)",
  "quota_remaining": "Number",
  "quota_limit": "Number",
  "period_start": "String (ISO date)"
}
```

**Redis keys:**
- `quota:used:{tenantId}` — Counter for quota used
- `tx:{txId}:status` — Transaction completion status
- `tx:{txId}:step:{stepName}` — Individual step completion

**Cassandra - audit_logs table:**
```sql
CREATE TABLE audit_logs (
  tx_id UUID,
  tenant_id text,
  action text,
  quota_before int,
  quota_after int,
  timestamp timestamp,
  PRIMARY KEY (tx_id, timestamp)
) WITH CLUSTERING ORDER BY (timestamp DESC);
```

## Test Plan

| Test Case | Expected Behavior |
|-----------|-------------------|
| Normal consumption | All three stores update; tx marked complete |
| Quota exhausted | DynamoDB rejects; returns 429; no Redis/Cassandra writes |
| Redis failure mid-transaction | DynamoDB compensates (restores quota); transaction fails |
| Cassandra failure | DynamoDB + Redis compensate; DLQ receives message |
| Duplicate request (same txId) | Returns cached success result; no side effects |
| Network timeout during Cassandra | DLQ processes; eventual consistency resolves |

## Conclusion

Cross-store writes don't require distributed transactions if you embrace idempotency from the start. By:

1. **Using deterministic transaction IDs** for deduplication
2. **Ordering writes** from most to least critical
3. **Implementing compensating transactions** for failure recovery
4. **Leveraging dead-letter queues** for permanent failures

You get a system that's resilient, auditable, and correct—even when networks fail.

The pattern above has handled millions of quota operations in production. It's not about eliminating failure—it's about making failure uneventful. Your users never notice, and your on-call engineer sleeps soundly.
