> "At 10 million daily active users spread across six continents, every millisecond of latency costs us roughly $1.2 million in annual revenue. Our caching layer isn't just an optimization—it's the backbone of our customer experience."

That was the message from our VP of Platform Engineering during theArchitecture Review Board meeting that kicked off Project Thunder. We were about to rebuild our entire caching infrastructure from the ground up.

## The Problem: Why Our Legacy Cache Fails at Scale

Our existing single-layer Redis cache was choking under peak loads. Black Friday 2025 saw P99 latencies spike to 380ms for product catalog reads—unacceptable for an e-commerce platform where page load time directly correlates with conversion rate. More critically, we had a single point of failure: one Redis cluster in us-east-1 serving global traffic, with readonly replicas in other regions that were frequently stale.

The engineering challenge was clear: design a caching system achieving **99.99% availability**, sub-100μs latency for the hottest 1% of products, and strong consistency within each data center while surviving complete DC failures.

## The Architecture: Three-Tier Caching Strategy

Our solution implements a three-tier caching architecture optimized for different access patterns:

```
┌─────────────────────────────────────────────────────────────────┐
│                     CDN Edge Network                          │
│  (200+ Points of Presence for static assets - images, CSS)   │
├─────────────────────────────────────────────────────────────────┤
│                   L1: Local Caffeine Caches                    │
│  (In-process caches per service instance, 50MB heap each)     │
├─────────────────────────────────────────────────────────────────┤
│               L2: Redis Cluster (per Data Center)               │
│  (6-node cluster per DC, 3 replicas + 3 shards)               │
├─────────────────────────────────────────────────────────────────┤
│              Cross-DC Replication via Kafka                     │
│         (Event-driven eventual consistency)                  │
└─────────────────────────────────────────────────────────────────┘
```

### Why Three Layers?

The cost of a cache miss in our system is significant. A product detail page makes roughly 12-15 downstream calls to gather all required data—pricing, inventory, reviews, recommendations. Each miss ripples through our entire backend.

**Layer 1 (Caffeine)** gives us in-process cache hits at sub-10μs. Since 80% of our traffic targets just 2% of our catalog (the hot products), local caching captures enormous value. A Caffeine cache with a 50MB heap and 5-minute TTL captures most buy-box sessions within a single container instance.

**Layer 2 (Redis Cluster)** provides shared caching across service instances within a DC. Redis gives us structured data types (Sorted Sets for ranked recommendations, Hashes for product catalogs) and mature cluster sharding. Our 6-node configuration (3 masters + 3 replicas, distributing across 3 availability zones) delivers the availability target.

**Layer 3 (CDN)** handles static content—product images, category thumbnails, promotional assets. This traffic is entirely cacheable at edge nodes and doesn't need L1/L2 complexity.

## Implementation: Cache-Aside with Probabilistic Early Expiration

We use the **cache-aside pattern** with a critical enhancement: **probabilistic early expiration (PEE)** to prevent thundering herd.

```java
@Service
public class ProductCacheService {

    private final CaffeineCache<String, Product> localCache;
    private final RedisTemplate<String, Product> redisTemplate;
    private final KafkaProducer<String, ProductUpdate> kafkaProducer;

    @Value("${cache.local.max-size:50000}")
    private int localMaxSize;

    @Value("${cache.local.ttl-minutes:5}")
    private int localTtlMinutes;

    @PostConstruct
    public void init() {
        this.localCache = Caffeine.newBuilder()
            .maximumSize(localMaxSize)
            .expireAfterWrite(localTtlMinutes, TimeUnit.MINUTES)
            .recordStats()
            .build();
    }

    public Optional<Product> getProduct(String sku) {
        // L1: Check local Caffeine cache
        Product product = localCache.getIfPresent(sku);
        if (product != null) {
            metrics.increment("cache.l1.hit");
            return Optional.of(product);
        }

        // L2: Check Redis
        product = redisTemplate.opsForValue().get(cacheKey(sku));
        if (product != null) {
            metrics.increment("cache.l2.hit");
            localCache.put(sku, product);

            // Probabilistic early expiration to prevent stampede
            if (shouldProbabilisticExpire(product)) {
                localCache.invalidate(sku);
            }
            return Optional.of(product);
        }

        metrics.increment("cache.miss");
        return Optional.empty();
    }

    private boolean shouldProbabilisticExpire(Product product) {
        // Randomly expire 1% of entries early to distribute load
        // during TTL renewal windows
        return product.getAccessCount() > THRESHOLD
            && random.nextDouble() < 0.01;
    }
}
```

The magic of probabilistic early expiration: instead of 10,000 containers expiring the same product's cache entry simultaneously at TTL boundary, we scatter expirations across a time window. The hot product stays in L1 cache almost continuously, but the aggregate load on Redis stays manageable.

## Write-Through: Strong Consistency Within a Data Center

For writes, we implement **write-through caching**:

```java
@Service
public class ProductWriteService {

    public void updateProduct(Product product) {
        // Atomic write to Redis first (source of truth)
        redisTemplate.opsForValue().set(cacheKey(product.getSku()), product);

        // Invalidate local caches across all instances
        localCache.invalidate(product.getSku());

        // Emit event for cross-DC replication
        kafkaProducer.send("product-updates", new ProductUpdate(
            product.getSku(),
            product,
            System.currentTimeMillis()
        ));

        metrics.increment("product.updated");
    }
}
```

Write-through ensures reads never see stale data within a DC. The Redis write commits before we return success—the local cache invalidation and Kafka event are fire-and-forget for the API response.

## Cross-DC Replication: Eventual Consistency via Kafka

Cross-DC replication is where eventual consistency becomes necessary. A true write-across-all-DC strategy would require synchronous quorum across continents—unacceptably slow. Instead, we use **async event-driven replication**:

```java
@Configuration
public class KafkaReplicationConfig {

    @Bean
    public KafkaConsumer<String, ProductUpdate> crossDcConsumer() {
        var consumer = new KafkaConsumer<>(Map.of(
            "bootstrap.servers", "kafka-us-east-1.internal:9092",
            "group.id", "cross-dc-replication",
            "auto.offset.reset", "earliest"
        ));
        consumer.subscribe(List.of("product-updates"));
        return consumer;
    }
}

@Service
public class CrossDcReplicationService {

    @KafkaListener(topics = "product-updates", groupId = "cross-dc-replication")
    public void replicateToLocalDc(ProductUpdate update) {
        // Apply to local DC's Redis cluster
        redisEuWest1.opsForValue().set(
            cacheKey(update.getSku()),
            update.getProduct()
        );
        // Invalidate local L1 caches via Redis Pub/Sub
        redisEuWest1.convertAndSend("cache-invalidate", update.getSku());

        metrics.increment("cross-dc.replicated");
    }
}
```

This gives us **eventual consistency within 100-500ms** across DCs—far tighter than the 30-second to 5-minute replication windows we observed with our previous batch-sync approach.

## Handling Data Center Failure

Our 99.99% availability target requires surviving DC failures gracefully:

1. **Multi-AZ Redis deployment**: Each 6-node cluster spans 3 availability zones. Losing a single AZ doesn't trigger failover.
2. **DNS-based failover with health checks**: Route53 health checks detect unhealthy clusters; traffic redirects to healthy DCs within 30 seconds.
3. **Local read fallback**: If Redis is unreachable but the service is running, L1 Caffeine caches serve stale data with a `StaleWhileRevalidate` header for UI display.
4. **Circuit breakers**: When downstream services fail, we fail open at the cache layer rather than cascading failure.

In production testing, we've demonstrated recovery from complete DC failure in under 45 seconds with zero apparent outage to end users.

## Results and What We Learned

Nine months after launching Project Thunder:

- **P50 latency for hot products**: 12μs (down from 85ms)
- **P99 latency**: 28ms (down from 380ms)
- **Cache hit rates**: L1 at 67%, L2 at 94%, CDN at 99.8%
- **Availability**: 99.991% over the last quarter
- **Black Friday 2026**: Peak of 142,000 requests/second with P99 of 31ms

The key lessons? Layer your caches for the access patterns you actually have. Don't optimize for the cold 98%—optimize for the hot 2%. And always, always design for failure at the infrastructure layer before you need it.
